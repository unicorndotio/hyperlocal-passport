# ADR-004: Extend Existing Validate Endpoint with Optional Quantity Field

## Status

Accepted

## Date

2026-06-12

## Context

The checkout validation API at `POST /api/transactions/validate` currently accepts `{ code: string, amountCents: number }`. For BOGO and item-specific coupons, the cashier enters a quantity consumed rather than a total amount. The system knows the unit price from the coupon and should calculate both the discount and the total purchase value.

The team debated three approaches:
- **Extend existing endpoint** with an optional `quantity` field — when `quantity` is provided, `amountCents` is derived server-side
- **New separate endpoint** for quantity-based validation
- **Always pass amountCents** — cashier enters total for all types, system cross-validates

Analysis: A single endpoint is simpler to maintain and the behavior difference (total input vs quantity input) is already determined by the coupon's behavior type, which the server knows. Adding an optional field is minimal API surface increase. A separate endpoint duplicates validation logic.

## Decision

Extend `POST /api/transactions/validate` to accept an optional `quantity` field.

**New request shape:**
```json
{
  "code": "ABC123",
  "amountCents": 10000,
  "quantity": 6
}
```

**Behavior:**
- If `behavior.type` is `percentage_discount` or `fixed_amount`: `quantity` is ignored, `amountCents` is used as-is (existing behavior)
- If `behavior.type` is `bogo` or `item_specific`: `quantity` is required, `amountCents` is **calculated server-side** as `unitPrice * quantity`. If the client sends `amountCents`, it is validated against the calculated value for consistency but the calculated value takes precedence for the transaction record.

## Alternatives Considered

### New separate endpoint (`POST /api/transactions/validate-item`)

- **Pros**: Clean API contract per use case; no conditional logic in the handler
- **Cons**: Duplicate auth, validation, and response logic; two endpoints to maintain; the cashier client must know which endpoint to call based on coupon type
- **Why rejected**: The coupon type is already known server-side after looking up the redemption. Dispatching to the right calculation logic in one handler is simpler than dispatching at the API gateway level.

### Always pass amountCents (cashier enters total for all types)

- **Pros**: Uniform API contract; no UI changes for existing businesses
- **Cons**: For BOGO/item-specific, asking for the total amount instead of quantity consumed breaks the UX (cashier has to calculate the total instead of just inputting the count); loses the item-level data needed for resident savings tracking
- **Why rejected**: The PRD specifically requires quantity-based input for BOGO/item-specific to accurately track per-unit discounts and enable resident savings aggregation.

## Consequences

### Positive

- Single endpoint, single auth check, single validation pipeline
- Cashier UI varies by coupon type but the API contract remains consistent
- Server-side amount calculation ensures data integrity — the logged total purchase value is always correct

### Negative

- The handler now branches on behavior type, increasing cyclomatic complexity
- Request validation must conditionally require `quantity` based on the resolved coupon type

### Risks

- Client sends both quantity and amountCents that disagree. Mitigated by always using the server-calculated value for BOGO/item-specific; logging a warning if mismatch exceeds a threshold.
- Existing clients sending requests without `quantity` continue to work unchanged — no breaking change.
