# ADR-004: Immediate Business Access with Feature Gating During Activation

## Status

Accepted

## Date

2026-06-10

## Context

The PRD defines F5 — Business Self-Service Registration — where businesses register themselves and need to be activated by admin after payment confirmation (billed externally). The question is whether businesses should have any access before admin activation.

## Decision

Businesses receive immediate login access after registration but their coupons are only visible in the public catalog after admin enables the business. This allows businesses to set up their profile and create coupons in advance, so the moment admin activates them, everything is ready.

## Alternatives Considered

### Alternative 1: No access until admin activates

- **Description**: Business registers and can only log in after admin toggles active=true. No profile editing or coupon creation possible before activation.
- **Pros**: Simplest mental model — business sees nothing until they pay. No risk of content creation before payment.
- **Cons**: Wastes the time between registration and payment confirmation. Business must wait to set up profile. Admin must remind them to log in.
- **Why rejected**: Creates unnecessary friction. Letting businesses prepare their profile and coupons during the payment processing window reduces time-to-live for their listing.

### Alternative 2: Full access after registration, admin only for payment issues

- **Description**: Business is immediately active in the catalog. Admin only disables if payment fails.
- **Pros**: Fastest time-to-live. No admin bottleneck.
- **Cons**: Risk of businesses using the platform without paying. Requires admin to proactively detect non-payment.
- **Why rejected**: Without in-app payment, admin must have a positive enable action to confirm payment was received.

### Alternative 3: Immediate access + feature gating (chosen)

- **Description**: Business can log in, edit profile, and create coupons immediately. Coupons hidden from public catalog until admin enables the business. Admin toggle controls catalog visibility.
- **Pros**: Businesses can prepare everything in advance. Catalog only shows paying businesses. Simple admin mental model (toggle = payment received).
- **Cons**: Slightly more complex UI (must show "waiting for activation" state in business dashboard).

## Consequences

### Positive

- Businesses can fully set up their presence before payment is processed
- Catalog never shows non-paying businesses
- Admin activation is a single toggle with clear effect
- Reduces time between payment and live listing

### Negative

- Business dashboard needs an activation status indicator
- Slightly more state to handle in the UI (pending vs active vs inactive)
- Business may be confused about why their coupons aren't visible

### Risks

- Business confusion: Mitigation — show clear banner "Your listing is pending activation. You'll be live once your subscription is confirmed." Include instructions for payment.

## Implementation Notes

- The `isActive` field on Business controls catalog visibility
- New businesses are created with `isActive: false` by default
- Business dashboard checks `isActive` and shows banner if false
- Coupon CRUD does NOT depend on `isActive` — businesses can create coupons while pending
- Catalog endpoint filters by `isActive: true`
- Admin toggle changes `isActive` — this is the payment gate
