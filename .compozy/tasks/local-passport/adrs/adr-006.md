# ADR-006: Docker Containerization and Local File Storage

## Status

Accepted

## Date

2026-05-21

## Context

We need to store user-uploaded identity documents and business logos. Initially,
in ADR-003, we decided to use DigitalOcean Spaces (S3-compatible object
storage). However, for a lower-cost setup, simpler maintenance, and
self-contained deployments, running the application inside a Docker container
and storing uploaded files on the local filesystem using a persistent Docker
volume is a much more cost-effective and practical alternative.

## Decision

We will containerize the Passaporte Local Deno Fresh application using Docker,
following development best practices.

Specifically, we will:

1. Implement a local storage utility (`lib/storage.ts`) that writes files to the
   local filesystem under a configurable directory (e.g., `/app/uploads`), which
   lives outside the public static asset folder for security.
2. Implement a custom Fresh API route handler (`/api/uploads/[filename]`) to
   securely read, serve, and potentially restrict access to uploaded documents
   (Option B).
3. Prefix the stored document and logo file paths in Deno KV with a configurable
   `APP_BASE_URL` environment variable, ensuring the API returns full, valid
   absolute URLs.
4. Write a production-ready `Dockerfile` (using a specific Deno version,
   non-root user for security, caching Deno dependencies prior to building, and
   separating build steps if needed).
5. Write a `docker-compose.yml` file that mounts persistent Docker volumes:
   - One volume mapped to `/app/uploads` to store and persist uploaded documents
     and logos.
   - One volume mapped to `/app/data` to persist the Deno KV SQLite-backed
     database file (e.g., `/app/data/passport.db`).

## Alternatives Considered

### Alternative 1: DigitalOcean Spaces (S3 API compatible)

- **Description**: Storing uploaded files in DigitalOcean Spaces as initially
  decided in ADR-003.
- **Pros**: Highly scalable, offloads storage from the server, easily integrates
  with CDNs.
- **Cons**: Monthly service fee and potentially egress fees, adding external
  dependencies and credential management.
- **Why rejected**: The user wants a cheaper, self-hosted deployment option.

### Alternative 2: Saving files in ephemeral storage without volumes

- **Description**: Storing files locally on the container disk but without
  mounting any volumes.
- **Pros**: Simplest setup.
- **Cons**: Any container restart or redeployment would wipe all user documents
  and business logos.
- **Why rejected**: Data loss is unacceptable for a production-like or testing
  environment.

## Consequences

### Positive

- **Zero Operational Storage Cost**: No cloud object storage subscription is
  needed.
- **Self-contained deployment**: A single `docker compose up -d` spins up the
  app, database (Deno KV), and persistent storage volumes.
- **Offline compatibility**: Easier local testing and development since no cloud
  credentials or internet connection are required for uploads.

### Negative

- **Backup Management**: Backing up files and the Deno KV database is now the
  host server's responsibility (must back up host volumes).
- **Scalability Limit**: Limited by the disk space and I/O capabilities of the
  single hosting VM. If scaled to multiple server instances, we would need a
  shared volume (like NFS) or revert to object storage.
- **Security Warning**: Uploaded files are stored outside the public directory.
  However, since they are served via `/api/uploads/[filename]`, we can implement
  access control checks (e.g., verifying admin or owner credentials for identity
  documents). Business logos can remain publicly readable. Random UUID filenames
  will still be used to prevent path traversal or document enumeration.

## Implementation Notes

- The upload utility in `lib/storage.ts` will accept standard file/blob inputs,
  generate a unique UUID file name, save it to the target uploads folder, and
  return the filename.
- A custom API route handler (`routes/api/uploads/[filename].ts`) will read
  files from disk and serve them.
- In Deno KV, the file URLs will be constructed by combining the configurable
  `APP_BASE_URL` env var (e.g., `http://localhost:8000`) with
  `/api/uploads/<filename>`.
- Environment variables:
  - `UPLOADS_DIR` (defaults to `/app/uploads`)
  - `DENO_KV_PATH` (defaults to `/app/data/passport.db`)
  - `APP_BASE_URL` (defaults to `http://localhost:8000`)
