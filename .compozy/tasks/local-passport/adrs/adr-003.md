# ADR-003: Resident Demand Signals Storage and Notification

## Status

Accepted

## Date

2026-06-10

## Context

The PRD defines F7 — Resident Demand Signals — where residents can request services not yet in the catalog. These signals need to be stored, aggregated by category for admin review, and surfaced as notifications for unreviewed entries.

## Decision

Store individual demand signals as Deno KV entries with a timestamp and category, maintain a category-count index for quick aggregation, and show a badge/notification count on the admin dashboard for new unreviewed signals.

## Alternatives Considered

### Alternative 1: Flat KV list without indexing

- **Description**: Store all signals in a single KV key space without category indexes. Admin reads all entries and aggregates client-side.
- **Pros**: Simplest possible implementation, no write overhead.
- **Cons**: Every admin view requires scanning all entries; no easy count-by-category. Does not scale beyond a few hundred signals.
- **Why rejected**: Admin needs per-category counts to prioritize business recruitment. Scanning all entries on every admin page load is wasteful.

### Alternative 2: Store in external database (Postgres/SQLite)

- **Description**: Use a relational database for structured query needs (filtering, aggregation, pagination).
- **Pros**: Rich query capabilities, mature tooling for aggregation.
- **Cons**: Adds infrastructure dependency and schema management. The feature is simple enough for KV.
- **Why rejected**: Adding a database for a single simple feature violates YAGNI. KV indexes suffice for the expected scale (< 1000 signals/month in V1).

## Consequences

### Positive

- Signals are queryable by category for admin dashboards
- Admin can see new/unreviewed count at a glance
- No additional infrastructure beyond existing Deno KV
- Simple data model — easy to extend with reviewed/unreviewed status

### Negative

- Category-count index adds a small write overhead on each signal creation
- Pagination of raw signal list requires manual KV key design

### Risks

- If signals grow beyond thousands, the flat list scan for "all signals" may be slow. Mitigation: implement pagination with cursor-based KV iteration from the start.

## Implementation Notes

- KV key for individual signals: `["signals", "<signal_id>"]`
- KV key for category index: `["signals_by_category", "<category>", "<timestamp>"]` -> signal_id
- KV key for category counts: `["signal_counts", "<category>"]` -> number (updated atomically)
- Signal data: `{ id, category, description, residentId, createdAt, reviewed: boolean }`
- Admin dashboard: query category counts, display badge for sum of unreviewed, link to detailed view
