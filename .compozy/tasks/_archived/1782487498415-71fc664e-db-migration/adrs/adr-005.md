# ADR-005: Analytics Counters Model — Dedicated Table with Event-Based Sources

## Status

Accepted

## Date

2026-06-19

## Context

Coupon analytics track three metrics per coupon: views (page loads), redemptions (user generated a QR code), and validations (coupon used at checkout). Currently stored as separate KV atomic counter keys (`['analytics', couponId, metric]`). The PRD requires per-coupon funnel data for the business analytics dashboard.

Redemptions and validations already have their own event tables (`redemptions` and `transactions` tables respectively). Views are a simple page-load counter with no corresponding event record.

## Decision

Use a dedicated `coupon_analytics` table with one row per coupon, containing `views`, `redemptions`, and `validations` as integer columns. The `redemptions` and `validations` counters are updated atomically alongside redemption/transaction creation (using the same database transaction). The `views` counter is incremented via a simple UPSERT on business page load. For drill-down detail, the existing `redemptions` and `transactions` tables serve as the event log.

## Alternatives Considered

### Alternative 1: Separate Counter Tables Per Metric

- **Description**: Individual tables `coupon_views`, `coupon_redemptions`, `coupon_validations` with one row per event. Use SQL COUNT for aggregations.
- **Pros**: Fully normalized, supports time-range queries without extra columns.
- **Cons**: COUNT queries on large tables are slow without careful indexing. Each dashboard load requires three COUNT queries per coupon. More complex queries for the analytics API.
- **Why rejected**: The analytics dashboard loads per-coupon data frequently. Pre-aggregated counters provide O(1) reads.

### Alternative 2: SQL COUNT Only (No Aggregated Table)

- **Description**: Derive all analytics entirely from event tables via SQL COUNT queries. No `coupon_analytics` table.
- **Pros**: Zero redundant data, single source of truth.
- **Cons**: Views have no event table (page load counters are pure increments). Would require creating a `coupon_views` event table that grows unboundedly. COUNT queries on `redemptions` and `transactions` scale poorly at high volume.
- **Why rejected**: Views need a counter regardless. Pre-aggregated counters are simpler and faster for the dashboard.

## Consequences

### Positive

- Dashboard reads are O(1) — single row lookup per coupon
- Redemptions and validations counters stay in sync with their event records (updated in the same DB transaction)
- No redundant event table for views needed
- Simple schema: one row per coupon with three integer columns

### Negative

- Slightly more write work per redemption/validation (update analytics row + insert event row in same transaction)
- Counter drift possible if a redemption is rolled back but analytics counter isn't (mitigated by same-transaction updates)
- Views counter is an approximated counter (increment without deduplication — same as current KV behavior)

### Risks

- **Analytics counter drift**: If the database transaction succeeds but the analytics counter update is somehow skipped, counts drift from reality. *Mitigation:* Both the event row and analytics counter update are in the same Drizzle transaction. If one fails, both roll back.
- **View counter contention**: Multiple concurrent page views for the same coupon cause transaction retries on the analytics row. *Mitigation:* Use Drizzle's `sql` raw increment (`UPDATE coupon_analytics SET views = views + 1`) which is atomic and avoids read-modify-write contention.

## Implementation Notes

- Table: `coupon_analytics(id, coupon_id [FK], views [default 0], redemptions [default 0], validations [default 0])`
- Unique constraint on `coupon_id`
- Counter increments use SQL `SET views = views + 1` (atomic, no read-before-write)
- Row created on first view event (UPSERT pattern)
- Row deleted when coupon is deleted (CASCADE)
