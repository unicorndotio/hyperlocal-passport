# ADR-006: Test Database Strategy — Dedicated Test Database

## Status

Accepted

## Date

2026-06-19

## Context

All existing tests use `Deno.openKv(':memory:')` for isolated, fast database testing. With PostgreSQL, there is no built-in in-memory mode. A different isolation strategy is needed. Options include a dedicated test database, transaction-per-test rollback, or disposable containers.

The test suite has 34 files covering unit tests, integration tests, auth enforcement, and edge cases. The migration must preserve 100% test pass rate without slowing the test suite to an unusable degree.

## Decision

Use a dedicated `passport_test` PostgreSQL database. Tests connect to this database using the same Drizzle client pattern as production. Before each test run, the schema is reset (DROP + re-migrate + seed). Individual tests clean up their data via Drizzle DELETE/TRUNCATE in `beforeEach` hooks.

## Alternatives Considered

### Alternative 1: Transaction-Per-Test Rollback

- **Description**: Open a Drizzle transaction in `beforeEach`, run all test DB operations inside it, rollback in `afterEach`. 
- **Pros**: Fastest isolation — no cleanup overhead per test, no database state leaks.
- **Cons**: All queries under test must use the same transaction handle passed through the code. Current route handlers create their own Drizzle queries — they don't accept an external transaction handle. Refactoring all handlers to accept an optional transaction parameter is a significant change that contradicts the "no feature changes" constraint.
- **Why rejected**: The architectural change required (passing transaction handles through route handlers) is too invasive for a migration that should be a behind-the-scenes infrastructure change.

### Alternative 2: Testcontainers (Disposable PostgreSQL Container)

- **Description**: Use the `testcontainers` npm package to spin up a throwaway PostgreSQL container per test run. Fresh database every run.
- **Pros**: Cleanest isolation, no risk of cross-test contamination.
- **Cons**: Startup time of 5-15 seconds per test run. Docker-in-Docker complexity. `testcontainers` for Deno is not yet mature.
- **Why rejected**: Slow startup and unreliable Deno compatibility. A dedicated test database is simpler and faster.

## Consequences

### Positive

- No code changes to route handlers (no transaction handle passing needed)
- Test speed comparable to KV tests (single database, no container lifecycle)
- Simple setup: one environment variable (`PG_CONNECTION_TEST`) pointing to test database
- Tests can be run in parallel (each test file uses its own schema or prefix)

### Negative

- Test data persists between runs unless explicitly cleaned up — requires `beforeEach` cleanup hooks
- Test database must be created before running tests (manual step or in a setup script)
- Schema changes require re-running migrations on the test database
- Risk of test pollution if cleanup hooks are incomplete

### Risks

- **Test pollution from incomplete cleanup**: A test that inserts a user and forgets to delete it affects subsequent tests. *Mitigation:* Strict before-each cleanup that truncates all tables. Add a CI step that recreates the test database from scratch before each test run.
- **Slow test suite**: If cleanup is too aggressive (DROP/CREATE schema before each file), tests become slow. *Mitigation:* Truncate only the tables used by each test file, not the entire schema.

## Implementation Notes

- Environment: `PG_CONNECTION_TEST=postgresql://root:password@localhost:5433/passport_test`
- Test setup script: `deno task db:test:setup` — creates `passport_test` database and applies migrations
- Before each test file: `await db.delete(schema.table)` for tables used in that file
- Before each test run in CI: `DROP SCHEMA public CASCADE; CREATE SCHEMA public;` + re-run migrations
- Test files that test the same component share a cleanup strategy (e.g., all coupon tests truncate `coupons`, `redemptions`, `analytics` tables)
