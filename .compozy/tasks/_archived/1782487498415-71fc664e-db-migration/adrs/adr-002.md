# ADR-002: Schema-First, Drop and Recreate Data Migration Strategy

## Status

Accepted

## Date

2026-06-19

## Context

The migration from Deno KV to PostgreSQL requires moving from a schema-less key-value store to a relational database with a defined schema. There are two fundamental approaches: define schemas and create tables from scratch (schema-first), or create tables first then adapt schemas to match (database-first). Additionally, the question of whether to preserve existing KV data arises.

Since the application is in active local development with no production users, the existing KV data consists only of test/development records that can be safely discarded.

## Decision

Use a **schema-first approach**: define all Drizzle table schemas in TypeScript as the single source of truth, generate the initial migration with `drizzle-kit generate`, and apply it to create all tables. Drop the old Deno KV data file entirely — no data migration script is needed.

## Alternatives Considered

### Alternative 1: Database-First with Data Migration Script

- **Description**: Create PostgreSQL tables manually (or via `drizzle-kit pull` from an existing DB), then write a script to read all records from Deno KV and insert into PostgreSQL.
- **Pros**: Preserves existing development data, no data loss risk.
- **Cons**: Requires writing and maintaining a one-time migration script. Existing test data has no business value (it's all fake records). Schema would be designed around existing KV shapes rather than optimal relational design.
- **Why rejected**: No production users means no valuable data. A clean schema-first approach produces a better relational design without being constrained by KV's flat document shapes.

### Alternative 2: Dual-Write During Transition

- **Description**: Write to both KV and PostgreSQL during a transition period. Read from PostgreSQL once verified. Remove KV writes after confidence is established.
- **Pros**: Rollback possible at any point during transition.
- **Cons**: Every write operation becomes 2x the work. Significant code complexity with dual-database middleware. Overkill for pre-launch project.
- **Why rejected**: Pre-launch project with no production traffic does not need a gradual cutover.

## Consequences

### Positive

- Clean relational schema not constrained by KV document shapes
- No migration script to write or maintain
- Faster development — generate migration and immediately start coding against PostgreSQL
- Schema is the single source of truth — no drift between code and database

### Negative

- Any existing development data (test residents, businesses, coupons) is lost
- Seed script (`seed.ts`) must be updated to use Drizzle instead of direct KV writes
- Developers must recreate their local data after the migration

### Risks

- **Accidental data loss if someone has production data**: Mitigated by the explicit "no production users" status. Add a pre-migration checklist step to verify no real user data exists.
- **Seed script outage**: The seed script creates admin users for local development. It must be migrated to Drizzle before the team can develop after migration. *Mitigation:* Prioritize seed script migration alongside the schema definition.

## Implementation Notes

- Order of schema definition matters: tables with foreign key references must be defined after their referenced tables.
- The initial migration should be a single SQL file that creates all tables, indexes, and constraints.
- After migration, set `s.procedural` to `true` in local config to prevent accidental drops.
- The seed script output must be verified (admin user created, can log in, can access admin routes) before declaring migration complete.
