# ADR-007: Rate Limiting — Skipped for V1

## Status

Accepted

## Date

2026-06-19

## Context

The current codebase has two rate limiters for demand signal creation: `['ratelimit:signals', dateStr, userId]` (5 signals per day) and `['ratelimit:signals_hourly', hourStr, userId]` (3 signals per hour). These are stored as KV counter integers with no TTL — counters reset naturally when the key pattern changes (new date/hour).

The app has no real users and is being rebuilt before customer launch. Rate limiting is a production safety measure, not a core business feature. The PRD's goal is infrastructure modernization, not feature completion.

## Decision

Remove rate limiting from the signal creation endpoint entirely for V1. The rate limiter KV keys and their associated logic in the signal creation route will not be migrated to PostgreSQL. Rate limiting can be re-implemented when customer features are built and abuse is a real concern.

## Alternatives Considered

### Alternative 1: PostgreSQL-Based Rate Limiting Table

- **Description**: A table `rate_limits(resource, user_id, window_start, count)` updated on each signal creation. Background cleanup removes expired windows.
- **Pros**: Persisted across restarts, consistent with other data, queryable for analytics.
- **Cons**: Adds a database write to every signal creation (already the heaviest write path). Requires periodic cleanup jobs or cron. No real benefit until there are real users to rate-limit.
- **Why rejected**: Premature optimization. The table and cleanup logic would need to be designed, implemented, and tested for zero benefit at V1.

### Alternative 2: In-Memory Rate Limiting

- **Description**: A `Map` in the Deno process with TTL-based expiration. Fastest possible checks.
- **Pros**: No database round-trip, simple implementation.
- **Cons**: Counters reset on server restart. Multiple server instances (if scaled) share no state.
- **Why rejected**: Same problem — no real users to rate-limit. The in-memory approach would need to be replaced with a distributed solution when scaling, creating double work.

## Consequences

### Positive

- Less code to write and maintain for V1
- Simpler signal creation endpoint
- No migration work for rate limiter KV keys
- Faster signal creation (no counter write)

### Negative

- No abuse protection for the signal creation endpoint
- Must be implemented before customer launch in a future PRD

### Risks

- **No immediate risk**: The app has no public users. If a dev abuse scenario arises, the endpoint can be manually rate-limited at the reverse proxy (Caddy/nginx) level.

## Implementation Notes

- The two KV keys for rate limiting (`ratelimit:signals` and `ratelimit:signals_hourly`) are simply not migrated
- Signal creation handler removes the rate check: `routes/api/signals/create.ts` — delete the rate limit check block
- Rate limit KV utilities in `lib/ratelimit.ts` (if they exist as a separate module) — delete the file
- Rate limiting will be added back in a future customer features PRD
