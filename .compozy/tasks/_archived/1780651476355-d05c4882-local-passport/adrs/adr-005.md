# ADR-005: Authentication Strategy

## Status

Accepted

## Date

2026-05-17

## Context

A plataforma possui três perfis de usuários (Admin, Lojistas e Moradores) que
precisam ser autenticados com segurança. Era necessário escolher entre
implementar um sistema proprietário completo baseado em Deno KV, usar provedores
externos (como Clerk/Supabase), ou usar uma biblioteca flexível em código.

## Decision

Decidimos adotar a biblioteca **Better Auth** (https://better-auth.com/) para
gerenciar sessões, recuperação de senhas e autenticação de usuários. A
biblioteca é multi-framework, agnóstica de banco de dados e suporta nativamente
o ecossistema TypeScript moderno.

## Alternatives Considered

### Alternative 1: Deno KV Auth proprietário

- **Description**: Criar manualmente rotas de registro, verificação de hash
  (bcrypt/argon2) e gerenciamento de cookies JWT na unha.
- **Pros**: Zero dependências de terceiros.
- **Cons**: Muito propenso a erros de segurança; reinvenção da roda para funções
  básicas como "recuperação de senha".
- **Why rejected**: Consome tempo valioso de desenvolvimento que deveria ser
  focado nas regras de negócio.

### Alternative 2: Supabase Auth / Clerk

- **Description**: Usar um Identity Provider externo.
- **Pros**: Segurança de nível empresarial, telas prontas, gestão de risco de
  logins.
- **Cons**: Adiciona dependência de um serviço externo que possui limites em
  tiers gratuitos, além de espalhar dados de usuários entre a plataforma de auth
  e o banco local.
- **Why rejected**: Better Auth permite manter todo o controle "in-house"
  rodando na mesma infraestrutura sem depender de serviços externos que geram
  vendor lock-in e custos mensais extras no futuro.

## Consequences

### Positive

- Abstração completa dos fluxos complexos de autenticação.
- O ecossistema Better Auth é extensível e plugável (ex: suporta magic links,
  MFA, OAuth facilmente se decidirmos adicionar).

### Negative

- É necessário garantir que existe um adaptador de banco de dados (database
  adapter) do Better Auth compatível com o Deno KV (ou escrever um adaptador
  simples mapeando as chamadas para o KV).

## Implementation Notes

- O Better Auth precisará ser injetado nas rotas de middleware do Deno Fresh
  para proteger `/api/*` e páginas privadas.
- A base de usuários e sessões gerada pelo Better Auth residirá nativamente
  dentro do nosso Deno KV.
