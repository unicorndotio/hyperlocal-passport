# ADR-002: Backend API and Database Infrastructure

## Status

Accepted

## Date

2026-05-17

## Context

Para o MVP do Passaporte Local, precisávamos definir a infraestrutura de
backend. O frontend já está definido utilizando Deno Fresh. A aplicação requer
rotas de API para registrar moradores, aprovar perfis, e processar a validação
de descontos no caixa.

## Decision

Decidimos utilizar as **API Routes nativas do Deno Fresh** (`routes/api/*`)
juntamente com o **Deno KV** como banco de dados principal. O Deno KV oferece
consistência forte (globally distributed) e baixa latência, sem necessidade de
provisionamento de infraestrutura adicional.

## Alternatives Considered

### Alternative 1: Supabase (PostgreSQL)

- **Description**: Usar Supabase como Backend-as-a-Service com banco relacional.
- **Pros**: Consultas complexas SQL são mais fáceis; já inclui Auth e Storage.
- **Cons**: Adiciona uma dependência pesada de rede e complexidade de setup
  inicial maior do que o Deno KV nativo.
- **Why rejected**: Para manter a stack "edge-native" e maximizar a performance
  com a menor quantidade de peças móveis no MVP, o Deno KV é mais adequado.

## Consequences

### Positive

- Zero configuração de infraestrutura.
- Latência extremamente baixa nas rotas de API.
- Deploy simplificado no Deno Deploy.

### Negative

- Consultas complexas (como buscas textuais ou paginação complexa) exigem design
  cuidadoso das chaves (keys) no KV ou índices secundários manuais.

## Implementation Notes

- O design das chaves do Deno KV será crítico. Precisaremos estruturar prefixos
  de chaves para `users`, `businesses`, e `transactions`.
