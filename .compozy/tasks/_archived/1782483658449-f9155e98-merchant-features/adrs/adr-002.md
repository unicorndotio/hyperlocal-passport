# ADR-002: Business Dashboard Layout — Cohesive Redesign with Dedicated Analytics Tab

## Status

Accepted

## Date

2026-06-12

## Context

The merchant features workflow introduces a flexible coupon engine, template presets, coupon analytics, transaction history, and interactive onboarding. The current business dashboard has three tabs (Coupons, Checkout, Profile) with no analytics or transaction history views.

The team debated two layout approaches:
- **Approach A**: Redesign the dashboard with a 4th "Analytics" tab. Coupon creation and analytics are separated concerns. Transaction history lives under Analytics. Interactive onboarding on first login.
- **Approach B**: Keep the existing 3-tab layout. Analytics data appears inline on the Coupon Manager page; transaction history on the Checkout page. Tooltip-based lightweight onboarding.

Analysis: The current Coupon Manager page already shows a table of coupons with usage progress bars. Adding a full analytics funnel (views → redemptions → validations) inline would severely constrain the layout and compete with the creation form for vertical space. A dedicated tab gives each feature room to breathe and matches the pattern established by competitors (Fidelio, Grazie, Loopy all have separate analytics sections).

## Decision

Adopt Approach A: Cohesive dashboard redesign with 4 tabs — Coupons, Validar Cupom, Analytics, Perfil. Interactive onboarding (walkthrough) guides businesses through the new features on first login after the update.

## Alternatives Considered

### Alternative 1: Progressive Enhancement (Approach B)

- **Description**: Keep 3-tab layout, add analytics inline on Coupon Manager, transaction history on Checkout page
- **Pros**: Fewer navigation changes for existing businesses; less upfront design work
- **Cons**: Analytics competes for space with coupon creation; transaction history competes with validation form; cramped mobile layout
- **Why rejected**: Results in a compromised UX for both analytics and core tasks. Dedicated tab is the cleaner long-term pattern.

### Alternative 2: Minimal Viable Enhancement

- **Description**: Replace coupon form with new engine, skip analytics and transaction history in V1
- **Pros**: Fastest ship time
- **Cons**: Contradicts user's explicit requirement for analytics; no feedback loop for businesses
- **Why rejected**: Analytics is a core part of the value proposition — businesses need to see coupon ROI to stay subscribed.

## Consequences

### Positive

- Clean separation of concerns: create/manage coupons in one tab, measure impact in another
- Analytics tab has full screen space for funnel visualization
- Transaction history fits naturally under Analytics
- 4-tab pattern is extensible for future additions (e.g., Posts/Events in V2)
- Interactive onboarding reduces support burden from the new UI

### Negative

- Existing businesses need to learn the new tab layout
- More frontend work (new route, new island component for analytics)

### Risks

- Businesses may not discover the Analytics tab. Mitigated by interactive onboarding that highlights each tab.
- Analytics data may be sparse initially (few validations). Mitigated by showing helpful empty states and projected values.

## Implementation Notes

- New route: `/business/analytics` with an `AnalyticsDashboard` island
- BusinessHeader updated to show 4 tabs
- CouponManager island updated with template-based creation flow
- CheckoutCalculator island updated with multi-behavior discount calculation
- Onboarding walkthrough triggers once per business on first login after feature deploy (tracked via a `hasSeenMerchantOnboarding` flag)
- Transaction history: fetch from `['business_transactions', businessId, ...]` KV prefix, paginated
