# ADR-001: Coupon Engine Architecture — Discriminated Union with UI-Gated Behaviors

## Status

Accepted

## Date

2026-06-12

## Context

The current coupon system has two types (`basic` and `special`) that are behaviorally identical — only `discountPercent` is supported. The merchant-features workflow needs to support multiple coupon behavior types (Percentage Discount, Fixed Amount, BOGO, Item-Specific) combined with configurable restrictions (caps, time windows, usage frequency, scope).

The team debated between three architectural approaches:
- **Discriminated union** (architect-advisor): embed behavior+restrictions as a sealed TypeScript union in a single Coupon document, dispatch calculation via strategy pattern
- **Orthogonal condition×effect×budget** (the-thinker): decompose every coupon into three orthogonal axes — types are just presets
- **Incremental** (pragmatic-engineer + product-mind): ship a subset first, expand later

The user confirmed they want all behavior types, but the council identified UX overwhelm as a real risk for micro-entrepreneurs.

## Decision

1. **Architecture**: Use a discriminated union for coupon behaviors embedded in a single Coupon document. Each behavior variant has explicit, typed fields. Calculation dispatched via strategy pattern keyed on behavior type.
2. **V1 scope**: Ship the full restriction engine (Global Cap, User Cap, Valid From/Until, Usage Frequency, Max Units per Redemption, Application Scope). Ship all four behavior types in the codebase.
3. **Template presets**: Instead of gating behavior types, provide template presets (e.g., "Simple Discount", "Flash Sale", "Loyalty Perk", "Event Promo") that pre-fill restriction defaults. This simplifies the UI for micro-entrepreneurs while exposing full power when needed.
4. **Coupon analytics**: Include a basic analytics view showing coupon performance funnel (views → redemptions → validations) so businesses see ROI.
5. **Foundation fixes**: Add a proper `coupons_by_businessId_and_isActive` index with pagination. Add a coupon edit PATCH endpoint alongside the new engine.
6. **Opening hours fix**: Allow removing individual days from the opening hours UI (data model already supports it).

## Alternatives Considered

### Alternative 1: Orthogonal condition×effect×budget model

- **Description**: Coupons are decomposed into independent axes — activation conditions, discount effects, budget constraints. Types are named presets over the composition.
- **Pros**: More flexible for future permutations; no type-specific code paths; natural V2 expansion path
- **Cons**: Over-engineered for 4 known types; adds rule-engine complexity for no validated benefit; harder to debug and audit in V1
- **Why rejected**: Premature abstraction. Start with discriminated union, generalize to this model in V2 if usage data reveals orthogonal axes.

### Alternative 2: Ship subset of behaviors in V1

- **Description**: Launch with Percentage Discount + Fixed Amount only. Defer BOGO and Item-Specific to V2.
- **Pros**: Faster initial ship; less testing surface; simpler UX
- **Cons**: User explicitly wants all types; BOGO is a real use case for bars/restaurants; building the restriction engine twice costs more
- **Why rejected**: Mitigated by UI gating — full engine ships but only subset is visible. Businesses that need BOGO can be enabled via feature flag.

## Consequences

### Positive

- Single Coupon document = one KV read per lookup, no joins
- Discriminated union gives type safety and compile-time exhaustiveness checking
- Strategy pattern isolates complex calculation logic per behavior type
- UI gating lets us ship full engine without overwhelming micro-entrepreneurs
- Feature flags let us roll out BOGO/Item-Specific to specific businesses for beta testing

### Negative

- Full restriction matrix in V1 has larger testing surface
- Must maintain feature-flagged code paths until behaviors are promoted to GA
- O(N) scan for coupon listing must be addressed before or alongside this work

### Risks

- **KV-scale risk**: Embedding behavior+restrictions makes each Coupon document larger, slowing scans. Mitigated by indexed listing and pagination.
- **Migration risk**: Existing coupons have `discountPercent` but no behavior type. Need a migration script to auto-convert. Existing coupons become `{ behavior: { type: 'percentage_discount', percent } }`.
- **UX risk**: Even gated UI (Percentage + Fixed Amount + full restrictions) has more fields than current form. Mitigated by progressive disclosure in the UI — show basic fields first, "Advanced restrictions" collapsible section.

## Implementation Notes

- New Coupon type: `behavior` is a discriminated union with a `type` discriminant field
- Existing `discountPercent` field is removed; migrated to `behavior: { type: 'percentage_discount', percent: discountPercent }`
- Existing `globalLimit`, `userMonthlyLimit`, `validUntil` remain but move under a `restrictions` sub-object
- `globalClaimedCount` stays at coupon root for fast counter access
- Checkout calculator dispatches to `CouponEngine.calculate(behavior, restriction, cart)` implementing the strategy pattern
- Migration script: iterate `kv.list({prefix: ['coupons']})`, transform old format to new, re-write atomically
