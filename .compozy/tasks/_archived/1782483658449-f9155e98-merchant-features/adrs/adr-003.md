# ADR-003: Analytics Counters in Dedicated KV Prefix

## Status

Accepted

## Date

2026-06-12

## Context

The coupon engine needs counters for views, redemptions (currently `globalClaimedCount` on the Coupon document), and validations. These counters are read on every redeem and validate operation and incremented atomically.

The team debated three storage approaches:
- **Root field on Coupon document** — counter lives at the Coupon document root, read/written with the full document
- **Inside behavior/restrictions** — counter embedded in the discriminated union
- **Separate KV prefix** — counters in a dedicated `['analytics', couponId, counterName]` key pattern

Analysis: Storing counters on the Coupon document means every redemption changes the document's versionstamp, causing atomic check conflicts under concurrent writes. As the platform scales, this contention increases. A dedicated analytics prefix decouples operational coupon data (behavior, restrictions) from volatile counters. The Coupon document only changes when the business edits the coupon, not on every redemption.

## Decision

Store all analytics counters in a dedicated KV prefix: `['analytics', couponId, counterName]`.

- `['analytics', couponId, 'views']` — number of times the coupon was viewed on the business detail page
- `['analytics', couponId, 'redemptions']` — number of redemptions created (replaces `globalClaimedCount`)
- `['analytics', couponId, 'validations']` — number of validations at checkout (redundant with transaction records but provides fast counter access)

The Coupon document no longer tracks `globalClaimedCount`. The `globalLimit` check reads from `['analytics', couponId, 'redemptions']` instead.

## Alternatives Considered

### Root field on Coupon document

- **Pros**: Single KV read for all coupon data; same pattern as existing code; simpler atomic transactions
- **Cons**: Every redemption bumps the Coupon's versionstamp; concurrent redemptions cause atomic conflicts; Coupon document grows with every write
- **Why rejected**: The versionstamp contention on the coupon document is a scalability bottleneck. Decoupling counters eliminates this without adding meaningful complexity — the redeem endpoint already reads the coupon document separately.

### Separate counter per model type (`['coupon_redemption_count', couponId]`)

- **Pros**: Clear key naming
- **Cons**: Mixes analytics with operational namespaces; harder to extend when new metric types are added
- **Why rejected**: A dedicated `analytics` prefix provides a clean namespace for all future metrics.

## Consequences

### Positive

- Coupon document never changes from redemption traffic — no versionstamp contention on the coupon template
- Analytics keys can be read/written independently from coupon data
- Prefix pattern is extensible: `['analytics', couponId, 'metric']` supports any future metric without schema changes
- Atomic operations on analytics keys are smaller and faster than on the full Coupon document

### Negative

- Redeem endpoint now reads two KV keys (coupon + analytics count) instead of one
- Atomic transaction must check and update two keys instead of one — slightly more complex

### Risks

- Counter drift: if the atomic transaction increments the counter but fails before creating the redemption record, the counter could exceed actual redemptions. Mitigated by Deno KV atomic transactions — all operations within a transaction commit or roll back together.
- Hot key contention: a popular coupon's analytics key could be a write hotspot. Mitigated by the atomic nature of Deno KV — conflicts are detected and the operation is retried.

## Implementation Notes

- Remove `globalClaimedCount` from the Coupon interface
- Add analytics key builders in `lib/coupon.ts` or a new `lib/analytics.ts`
- Update redeem endpoint: read `['analytics', couponId, 'redemptions']` instead of `coupon.globalClaimedCount`
- Add view counter increment to the business detail page handler
- Validation endpoint: increment `['analytics', couponId, 'validations']` on successful validation
