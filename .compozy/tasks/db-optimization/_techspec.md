# Technical Specification — Database Reliability Standardization

## Executive Summary

Standardize Passaporte Local's database layer in a single Big Bang effort: migrate
app-owned table PKs from `text` to `uuid`, add explicit `onDelete` to all 15
foreign keys, switch all timestamps to `withTimezone: true`, add 4 missing
indexes, realign seed data, convert raw SQL in `lib/` to typed Drizzle queries,
create a shared test DB utility, add CI guardrails via a standalone conventions
checker, and produce a written conventions guide. Migration uses delete+regenerate
since no production data exists. The materialized view `feed_events` gets a
read-only `pgTable` mapping in schema.ts for typed queries without data
duplication.

**Key decisions:**
- Big Bang execution in a single branch (per ADR-002) with organized commits
- Delete+regenerate migration strategy (single fresh `drizzle-kit generate`)
- `pgTable` mapping for `feed_events` MV in schema.ts (typed query builder, no storage)
- `useDatabase()` test utility with automatic setup/teardown hooks
- Standalone Deno script for CI conventions checking

**Primary trade-offs:**
- Larger single PR increases review burden but produces one atomic before/after
- Mixed `text`/`uuid` convention between Better Auth and app tables adds minor
  cognitive overhead
- Seed data loses human-readable IDs but gains UUID consistency

## System Architecture

### Component Overview

| Component | Responsibility | Boundaries |
|-----------|---------------|------------|
| `db/schema.ts` | Source of truth for all table definitions | 11 Drizzle `pgTable` definitions + relations + MV mapping |
| `db/migrations/` | Auto-generated SQL migration files | Generated by `drizzle-kit generate`; reviewed manually |
| `lib/db.ts` | `pg.Pool` singleton + `drizzle()` client | Shared by all lib/ modules, used by seed.ts |
| `lib/test-db.ts` | Shared pool/test lifecycle helpers | Created in Layer 3; consumed by test files |
| `lib/feed.ts` | Feed query (MV + transaction savings) | Converted from mixed raw SQL to typed Drizzle |
| `lib/analytics.ts` | Coupon analytics (view count + queries) | UPSERT converted from raw SQL to typed Drizzle |
| `seed.ts` | Development seed data | Realigned to auto-generated UUIDs |
| `scripts/check-schema-conventions.ts` | CI conventions checker | Inspects `db/schema.ts` for rule compliance |
| `docs/DB_CONVENTIONS.md` | Written conventions guide | Plain Markdown, audience: current + future developers |

### Data Flow

```
Schema changes (schema.ts)
  │
  ├── drizzle-kit generate ──► migration SQL (db/migrations/)
  │
  ├── lib/db.ts ──► typed queries ──► PostgreSQL
  │                       │
  │                       ├── lib/feed.ts (typed Drizzle)
  │                       ├── lib/analytics.ts (typed Drizzle)
  │                       └── tests/ (via lib/test-db.ts)
  │
  ├── seed.ts (auto-generated UUIDs)
  │
  └── CI (deno task check)
        └── scripts/check-schema-conventions.ts
```

## Implementation Design

### Core Interfaces

#### Test DB Utility (`lib/test-db.ts`)

```ts
// Provides a pooled DB client with automatic TRUNCATE cleanup around each test.

import { db, closeConnection } from './db.ts'

export function useDatabase(): { db: typeof db; close: typeof closeConnection }

// Returns { db } — the shared drizzle client.
// Before each test suite: no-op (pool is global singleton).
// After each test suite: optional TRUNCATE via exported helper.
// PG_CONNECTION guard built in — skips tests when not set.
// sanitizeResources/sanitizeOps: false applied automatically.
```

#### CI Conventions Checker (`scripts/check-schema-conventions.ts`)

```ts
// Exit code 0 = all checks pass, non-zero = violations found.
// Checks run against db/schema.ts as source of truth:
//   1. App-owned tables use uuid('id').defaultRandom() — not text('id')
//   2. All timestamp columns use { withTimezone: true }
//   3. All .references() calls include explicit onDelete
// Outputs clear messages like:
//   "ERROR: Table 'businesses' uses text('id') — use uuid('id').defaultRandom()"
//   "ERROR: Table 'coupons' column 'created_at' missing withTimezone: true"
//   "ERROR: Table 'redemptions' FK 'business_id' missing onDelete"
```

#### Added Indexes (4 total)

```ts
// 1. redemptions(user_id, status) — for resident active-codes query
index('idx_redemptions_user_id_status').on(table.userId, table.status)

// 2. verification(identifier) — for Better Auth token validation

// 3. transactions(redemption_id) — for join performance

// 4. file_metadata(user_id) — for access control lookups
index('idx_file_metadata_user_id').on(table.userId)
```

#### Materialized View Mapping (read-only, in schema.ts)

```ts
// Purely for typed query builder — does NOT create/prevent the MV
export const feedEvents = pgTable('feed_events', {
  id: text('id').primaryKey(),
  type: text('type').notNull(),
  title: text('title').notNull(),
  description: text('description'),
  imageUrl: text('image_url'),
  businessId: text('business_id'),
  businessName: text('business_name'),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull(),
})
```

The MV itself stays defined in the migration SQL (`CREATE MATERIALIZED VIEW`).
No insert/update/delete is ever performed through this definition.

#### file_metadata FK Addition

```ts
// Current (no FK):
userId: text('user_id'),

// New (with FK):
userId: text('user_id').references(() => users.id, { onDelete: 'cascade' }),
```

### Query Layer Conversions

#### `lib/analytics.ts` — UPSERT Conversion

**Before (raw SQL):**
```ts
await db.execute(sql`
  INSERT INTO coupon_analytics (id, coupon_id, views)
  VALUES (${analyticsId}, ${couponId}, 1)
  ON CONFLICT (coupon_id)
  DO UPDATE SET views = coupon_analytics.views + 1
`)
```

**After (typed Drizzle):**
```ts
import { sql } from 'drizzle-orm'

await db.insert(schema.couponAnalytics)
  .values({ id: crypto.randomUUID(), couponId, views: 1 })
  .onConflictDoUpdate({
    target: schema.couponAnalytics.couponId,
    set: { views: sql`coupon_analytics.views + 1` },
  })
```

`getCouponAnalytics` stays as-is (already typed Drizzle).

#### `lib/feed.ts` — Feed Query Conversion

**1. MV query** — typed via `feedEvents` pgTable mapping:

```ts
import { feedEvents } from '../db/schema.ts'
import { lt, desc } from 'drizzle-orm'

// Before:
let mvSql = sql`SELECT * FROM feed_events`
if (cursorDate) {
  mvSql = sql`${mvSql} WHERE created_at < ${cursorDate}::timestamptz`
}
mvSql = sql`${mvSql} ORDER BY created_at DESC LIMIT ${pageSize}`
const mvResult = await db.execute(mvSql)

// After:
const mvQuery = db.select().from(feedEvents)
if (cursorDate) {
  mvQuery.where(lt(feedEvents.createdAt, cursorDate))
}
const mvRows = await mvQuery
  .orderBy(desc(feedEvents.createdAt))
  .limit(pageSize)
```

**2. Transaction query** — typed Drizzle join:

```ts
import { transactions, businesses } from '../db/schema.ts'
import { eq, and, lt, gte, desc } from 'drizzle-orm'

// After:
const txRows = await db.select({
  id: transactions.id,
  discountAppliedCents: transactions.discountAppliedCents,
  timestamp: transactions.timestamp,
  businessId: businesses.id,
  businessName: businesses.name,
})
  .from(transactions)
  .innerJoin(businesses, eq(businesses.id, transactions.businessId))
  .where(
    and(
      eq(transactions.userId, userId),
      gte(transactions.timestamp, ninetyDaysAgo),
      cursorDate ? lt(transactions.timestamp, cursorDate) : undefined,
    ),
  )
  .orderBy(desc(transactions.timestamp))
  .limit(pageSize)
```

**3. `refreshFeedView`** stays as raw SQL (MV refresh is DDL, not a query):
```ts
await db.execute(sql`REFRESH MATERIALIZED VIEW CONCURRENTLY feed_events`)
```

**4. FeedEvent type realignment** — `createdAt` field keeps `number` (millisecond
timestamps from `.getTime()`). Row mapping changes from `String(row.id)` to
`row.id` (typed).

#### Seed.ts Realignment

**Before** — hardcoded string IDs:
```ts
const bizId = 'seed-biz-central-cafe'
await upsertBusiness({ id: bizId, userId: businessUserId, ... })

const couponId1 = 'demo-coupon-10off'
await upsertCoupon({ id: couponId1, businessId: bizId, ... })

const redemption1Id = 'seed-redemption-r1'
await upsertRedemption({ id: redemption1Id, couponId: couponId1, ... })
```

**After** — IDs omitted, `defaultRandom()` generates them:
```ts
// Business — omit id, let defaultRandom() handle it
// But we need the generated ID for FK references.
// Approach: insert with returning or use explicit crypto.randomUUID().

// Option A (returning — requires db.insert().returning()):
const [biz] = await db.insert(schema.businesses).values({
  userId: businessUserId,
  name: 'Café Central',
  ...rest
}).returning({ id: schema.businesses.id })

// Option B (explicit — keeps deterministic reference):
const bizId = crypto.randomUUID()
await db.insert(schema.businesses).values({
  id: bizId,
  userId: businessUserId,
  name: 'Café Central',
  ...rest
})
```

Selected: **Option B** (explicit `crypto.randomUUID()`). This keeps FK references
deterministic within seed.ts, avoids returning churn, and matches the existing
pattern used by `merchant_posts` and `coupon_analytics`. The UUIDs are
auto-generated but known within the seed function scope.

### Test DB Utility (`lib/test-db.ts`)

```ts
// API surface:
import { db } from './db.ts'
import { useDatabase } from '../lib/test-db.ts'

useDatabase()  // Must be called at top level of test file

Deno.test({
  name: '...',
  // sanitizeOps and sanitizeResources auto-set to false
  // PG_CONNECTION guard auto-applied
  fn: async () => {
    // db is ready — no setup needed
    // Cleanup: useDatabase() provides automatic TRUNCATE or manual helpers
  },
})
```

Implementation approach:
- Export `useDatabase()` that wraps `Deno.test` configuration
- Applies PG_CONNECTION skip guard
- Sets `sanitizeOps: false, sanitizeResources: false`
- Provides `cleanupDatabase()` helper that truncates all app tables
  (in dependency order to avoid FK violations during truncate)

|-------|-------------|

## Development Sequencing

### Build Order (Layer Sequence)

```
Layer 1: Foundation
  ├── Change PK types text → uuid on 8 app tables
  ├── Add explicit onDelete to all 15 FKs
  ├── Add missing FK on file_metadata.user_id
  └── Verify: deno task test

Layer 2: Structure
  ├── Add withTimezone: true to all timestamps
  ├── Add 4 missing indexes
  ├── Update feed_events MV definition (::timestamptz → timestamptz)
  ├── Add $onUpdate(() => new Date()) to merchant_posts.updated_at
  └── Verify: deno task test

Layer 3: Data & Queries
  ├── Realign seed.ts to auto-generated UUIDs (crypto.randomUUID())
  ├── Convert lib/analytics.ts UPSERT to typed Drizzle
  ├── Convert lib/feed.ts queryFeed() to typed Drizzle
  │   ├── Add feedEvents pgTable mapping in schema.ts
  │   ├── MV query: typed via feedEvents table
  │   └── Transaction query: typed Drizzle join
  ├── Create lib/test-db.ts with useDatabase() helper
  └── Verify: deno task test; seed flow works end-to-end

Layer 4: Enforcement & Documentation
  ├── Create scripts/check-schema-conventions.ts
  ├── Wire into deno task check
  ├── Write docs/DB_CONVENTIONS.md
  ├── Run drizzle-kit generate → review migration SQL
  └── Verify: deno task check passes with 0 violations
```

### Technical Dependencies

- All layers depend on Layer 1 (PKs must be uuid before FKs are rewritten)
- Layer 3 depends on Layer 2 (timestamps must be correct before queries reference them)
- Layer 4 depends on Layers 1-3 being finalized (CI script final-states the schema)
- Migration generation must be last (captures final schema)
- No external dependencies or infrastructure changes

## Technical Considerations

### Key Decisions

| Decision | Rationale | Trade-offs | Alternatives |
|----------|-----------|------------|--------------|
| `pgTable` mapping for MV | Enables typed queries without data duplication | Schema.ts has one read-only entry | Raw SQL (loses type safety) |
| Delete+regenerate migration | Clean slate with no data to migrate | Loses migration history | Sequential ALTER TABLE (complex, unnecessary) |
| Standalone Deno script for CI | Simple, explicit, composable with `deno task check` | Requires manual invocation | Custom lint plugin (more complex to build/maintain) |
| `useDatabase()` with hooks | Eliminates boilerplate duplication across ~40 test files | One more import pattern to learn | Minimal inline helpers (less DRY) |
| `crypto.randomUUID()` in seed | Deterministic FK references within seed function | Manual ID management | `returning()` (more verbose, extra query per insert) |
| Cascade for ownership, Restrict for audit | Clear semantic distinction | Must be documented in conventions guide | All-cascade (orphan risk for audit records) |

### Known Risks

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| **Test regression from PK type change** | Medium | Run tests after each layer; test suite exercises all join paths |
| **Feed query behavior change** | Low | `queryFeed()` output shape preserved (same `FeedEvent[]` interface) |
| **CI script fragility** | Low | Regex-based checks are surface-level and predictable |
| **Migration SQL error** | Low | Review generated SQL before committing; no data means clean CREATE |
| **Test utility adoption friction** | Low | Utility is additive — existing test files can migrate incrementally |
