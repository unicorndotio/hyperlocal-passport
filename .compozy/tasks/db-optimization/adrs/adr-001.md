# ADR-001: Schema Reliability Standardization — Scope and Approach

## Status

Accepted

## Date

2026-07-01

## Context

The project's database schema has four hygiene gaps identified in a prior review:
timezone-naive timestamps (no `withTimezone`), missing referential actions on 12
of 15 foreign keys, incomplete index coverage, and mixed primary-key types across
app-owned and auth tables. Separately, the query layer in `lib/` mixes raw SQL with
typed Drizzle queries, increasing maintenance surface area.

The project is in dev stage with no production data — the database is frequently
recreated, making this the only window for a clean schema rebase without migration
debt.

## Decision

Deliver a full-sweep database reliability standardization covering all four schema
areas, plus query-layer cleanup and CI guardrails, in a single execution:

1. **Primary keys**: Switch app-owned tables from `text('id')` to
   `uuid('id').defaultRandom()`. Keep `text` on Better Auth tables (`user`,
   `session`, `account`, `verification`).
2. **Referential integrity**: Add explicit `onDelete` to all 15 foreign keys.
   Cascade for ownership chains, restrict for audit records. Add missing FK on
   `file_metadata.user_id`.
3. **Timestamp timezone**: Switch all `timestamp('col')` to
   `timestamp('col', { withTimezone: true })`. Update `feed_events` MV. Remove
   `::timestamptz` raw casts.
4. **Indexing**: Add `redemptions(user_id, status)`, `verification(identifier)`,
   `transactions(redemption_id)`, `file_metadata(user_id)`.
5. **Seed data**: Realign to auto-generated UUIDs via `defaultRandom()`.
6. **Query layer**: Convert raw SQL in `lib/feed.ts` and `lib/analytics.ts` to
   typed Drizzle query builder.
7. **CI guardrails**: Custom Deno script checking `withTimezone`, `onDelete` FK
   policy, and PK types; git diff on migration snapshots for drift.
8. **Mini-guide**: `DB_CONVENTIONS.md` documenting patterns for future
   contributors.

## Alternatives Considered

### Alternative 1: FKs + targeted indexes only

- **Description**: Skip UUIDs and timezone changes; ship only referential
  integrity and the `redemptions(user_id, status)` index.
- **Pros**: Smallest blast radius, quickest to review and ship.
- **Cons**: Misses the only window for clean PK migration; timezone bugs remain
  latent.
- **Why rejected**: The user explicitly wants all four areas done in one go
  given the dev-stage window, and council consensus was that the PK + FK cluster
  is genuinely interdependent.

### Alternative 2: Phased approach (indexes + timezone first, FKs + UUIDs later)

- **Description**: Ship independent changes separately to reduce per-PR risk.
- **Pros**: Smaller review surface, easier rollback of individual items.
- **Cons**: Indexes and timezone changes touch the same files as the PK/FK
  cluster; splitting them creates merge conflicts and disjointed release notes.
  Council partially converged that UUIDs and FKs form an interdependent cluster.
- **Why rejected**: At dev stage, the cost of bundling (one larger migration) is
  outweighed by the benefit of a single atomic rebase.

## Consequences

### Positive

- One atomic migration captures all structural changes with zero migration debt
- CI guardrails prevent future drift and ensure new code follows conventions
- Query layer standardization eliminates raw SQL maintenance burden
- Mini-guide reduces onboarding time for new contributors
- All timestamps are timezone-aware, preventing a class of date-related bugs

### Negative

- Larger single PR increases review burden
- Mixed `text`/`uuid` convention (auth vs app tables) adds minor cognitive
  overhead for developers joining joins across the boundary
- Seed data loses human-readable IDs (e.g., `'seed-biz-central-cafe'`)

### Risks

- **PK type cascade**: UUID changes ripple through every FK definition; any
  misordered or mistyped column in the migration breaks the entire schema.
  Mitigation: since no data exists, migration is a clean CREATE — no ALTER TABLE
  gymnastics, just validate the generated SQL.
- **Query layer regression**: Converting raw SQL to typed Drizzle may uncover
  latent behavior differences. Mitigation: existing tests exercise the main query
  paths; run `deno task test` before and after.
- **CI script fragility**: Custom AST inspection on schema.ts may break as Drizzle
  syntax evolves. Mitigation: keep checks focused (regex for known patterns,
  not full AST parsing) and test the script in CI itself.

## Implementation Notes

- Order of operations: PK types → FKs (interdependent cluster), then indexes +
  timezone (independent, can be interleaved), then seed + query layer, then CI +
  guide.
- Migration: `drizzle-kit generate` produces a single CREATE migration.
  The generated SQL must be reviewed before applying. No ALTER TABLE since no
  existing data.
- Seed: replace hardcoded ID strings with `undefined` for `defaultRandom()` to
  fire, or use explicit `crypto.randomUUID()` if deterministic IDs are needed.
- CI script path: `scripts/check-schema-conventions.ts`, run via `deno task check`.
