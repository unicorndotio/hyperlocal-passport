# ADR-006: Standalone CI Conventions Script

## Status

Accepted

## Date

2026-07-01

## Context

After the schema standardization sweep, the project needs automated enforcement
to prevent future drift. Three patterns must be guarded: PK types (uuid for
app-owned tables), timestamp timezone setting (`withTimezone: true`), and FK
referential actions (explicit `onDelete`).

The enforcement mechanism runs as part of `deno task check` (which currently
runs `fmt --check + lint + deno check`). Two approaches exist: a standalone
Deno script that inspects `db/schema.ts`, or a custom Deno lint plugin.

## Decision

Create `scripts/check-schema-conventions.ts` â€” a standalone Deno script that:

1. Reads `db/schema.ts` as a text file
2. Uses targeted regex rules to check conventions:
   - App-owned table PKs must use `uuid('id').defaultRandom()` â€” not `text('id')`
   - All `timestamp(` calls must include `{ withTimezone: true }`
   - All `.references()` calls must include `{ onDelete:`
3. Outputs clear, actionable error messages per violation
4. Exits with code 0 (pass) or non-zero (fail, halts CI)

The script is wired into `deno.json` as an additional step in the `check` task:
`"check": "deno run -A scripts/check-schema-conventions.ts && deno fmt --check . && deno lint . && deno check"`

Better Auth tables (`user`, `session`, `account`, `verification`), the `feedEvents`
pgTable mapping, and `drizzle-orm` imports are excluded from PK type checking.

## Alternatives Considered

### Custom Deno lint plugin

- **Description**: Write a custom `deno lint` rule using the Deno lint plugin API
  to inspect the AST of schema.ts.
- **Pros**: Integrated into lint output; can leverage AST for more precise
  matching.
- **Cons**: Significantly more complex to build and maintain; lint plugin API is
  less commonly documented; the conventions are surface-level enough for regex.
- **Why rejected**: The user preferred the simpler standalone script. The rules
  are regex-friendly (matching `text('id')`, `timestamp(`, `.references()`), so
  a full AST parser is unnecessary.

### Git hooks only

- **Description**: Rely on pre-commit or pre-push hooks to catch violations
  instead of CI.
- **Pros**: Catches issues earlier in the developer workflow.
- **Cons**: Hooks are opt-in and can be bypassed; does not protect the main
  branch from non-compliant merges; the PRD mandates CI guardrails (F9).
- **Why rejected**: PRD F9 requires CI integration, not just git hooks.

## Consequences

### Positive

- Automated enforcement prevents schema drift after the sweep
- Clear, actionable error messages reduce debugging time
- Standalone script is easy to test and maintain
- Wired into existing `check` task â€” no new CI pipeline steps

### Negative

- Regex-based checks may produce false positives for unusual patterns
  (mitigation: test the script against known valid/invalid schema variants)
- The check runs on `db/schema.ts` text, not the AST â€” if Drizzle syntax evolves
  significantly, the regex patterns may need updating
