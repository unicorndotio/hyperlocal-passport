# ADR-003: pgTable Mapping for feed_events Materialized View

## Status

Accepted

## Date

2026-07-01

## Context

The `feed_events` materialized view is queried by `lib/feed.ts` for the hybrid
home feed. Currently, the view is created via raw SQL in the migration file and
queried via raw SQL with `::timestamptz` casts and manual row mapping
(`String(row.id)`, `new Date(row.created_at)`).

Drizzle ORM's typed query builder (`db.select().from()`) cannot reference the MV
directly because it is not a Drizzle `pgTable`. Two approaches exist: continue
using raw SQL for MV queries, or define a read-only `pgTable` that maps the MV's
columns for typed query builder access.

The concern about duplication is valid but addresses storage, not metadata:
a `pgTable` definition in schema.ts does not create a table, insert rows, or
duplicate data — it only provides column type information to the query builder.

## Decision

Add a read-only `pgTable` definition for `feed_events` in `db/schema.ts`:

```ts
export const feedEvents = pgTable('feed_events', {
  id: text('id').primaryKey(),
  type: text('type').notNull(),
  title: text('title').notNull(),
  description: text('description'),
  imageUrl: text('image_url'),
  businessId: text('business_id'),
  businessName: text('business_name'),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull(),
})
```

This definition:
- Is never inserted, updated, or deleted through Drizzle
- Does not create the MV — that stays in the migration SQL (`CREATE MATERIALIZED VIEW`)
- Is purely for the query builder to know column names and types
- Removes the need for `::timestamptz` casts and manual `String(row.id)` mapping

## Alternatives Considered

### Keep raw SQL with sql template literals

- **Description**: Continue using `db.execute(sql\`SELECT * FROM feed_events\`)` with
  manual result row mapping.
- **Pros**: No schema.ts additions; simpler to implement.
- **Cons**: No type safety on column names; manual type casting required; inconsistent
  with the goal of eliminating raw SQL from lib/.
- **Why rejected**: The PRD goal F6 explicitly requires typed Drizzle throughout lib/.
  The pgTable mapping is the standard Drizzle pattern for querying non-table objects.

### Define MV as a view in the migration only

- **Description**: Keep the MV definition in SQL only; accept raw queries for the
  MV part while converting only the transaction part of feed.ts to typed Drizzle.
- **Pros**: Minimizes schema.ts changes.
- **Cons**: Inconsistent half-converted state in feed.ts; the MV query remains
  untyped.
- **Why rejected**: The user preferred full consistency via the pgTable mapping.

## Consequences

### Positive

- Full type safety on MV queries — column name typos caught at compile time
- Consistent with the rest of lib/ (all typed Drizzle after Layer 3)
- No storage or data duplication — the table definition is metadata only
- Removes `::timestamptz` casts and manual row mapping

### Negative

- One extra export in schema.ts that is not a real table (minor documentation tax)
- Developers must understand this is read-only — insert/update/delete will fail
  at runtime against a materialized view
