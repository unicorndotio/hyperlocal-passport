# ADR-002: Big Bang Execution Strategy

## Status

Accepted

## Date

2026-07-01

## Context

The Database Reliability Standardization initiative covers 8 features across 4
schema areas (PK types, referential integrity, indexes, timezone handling) plus
query layer cleanup, CI guardrails, and a conventions guide. The work requires
changes to `db/schema.ts`, `seed.ts`, `lib/feed.ts`, `lib/analytics.ts`, test
fixtures, and the addition of new files (CI script, mini-guide).

Three execution strategies were considered: Big Bang (all at once), Incremental
Rollout (3 independent phases), and Foundation First (schema first, then
enforcement). The user has no fixed timeline and prioritizes quality over speed.

## Decision

Deliver all 8 features as a single focused effort — the Big Bang approach.
Work through the dependency chain sequentially within the same branch:

1. PK types (`text` → `uuid`) — foundational, everything else references this
2. Referential integrity — FK definitions depend on final PK types
3. Indexes + timezone — independent but interleaved between FK and query work
4. Seed data realignment — adapt to auto-generated UUIDs
5. Query layer standardization — convert raw SQL to typed Drizzle
6. CI guardrails — schema lint script + git diff check
7. Migration generation — single `drizzle-kit generate`
8. Mini-guide — `DB_CONVENTIONS.md`

## Alternatives Considered

### Alternative 1: Incremental Rollout

- **Description**: 3 independent phases — (1) indexes + timezone, (2) PKs + FKs,
  (3) query layer + CI + guide. Each phase shipped separately.
- **Pros**: Smaller reviews, independently deployable, easy to pause.
- **Cons**: Schema in mixed state between phases; risk of later phases drifting
  unscheduled; merge conflicts from other work touching schema mid-stream.
- **Why rejected**: The PK/FK cluster is genuinely interdependent and touching
  the same files as other changes, making clean separation difficult. No
  timeline pressure makes the "pause" advantage moot.

### Alternative 2: Foundation First

- **Description**: Schema sweep (PKs, FKs, indexes, timezone, seed, migration)
  as milestone 1. Query layer + CI + guide as milestone 2.
- **Pros**: Clear separation of structural vs procedural changes.
- **Cons**: Two larger PRs instead of one; milestone 2 requires self-discipline
  to ship.
- **Why rejected**: The user preferred single focused effort over splitting;
  Big Bang produces one atomic "before/after" moment in git history.

## Consequences

### Positive

- Single atomic migration with consistent state across all tables
- One "before/after" moment in git history — easy to understand and roll back
  if needed
- Dependency ordering ensures no schema contradictions during the transition
- No risk of phase 2 never shipping

### Negative

- Large review surface area — single PR touches many files
- Harder to parallelize across multiple contributors
- If blocked mid-stream, nothing is deployed until everything is done

### Risks

- **Review fatigue**: Single large PR may receive less thorough review per file
  than smaller focused PRs. Mitigation: organize commits within the branch as
  clean, reviewable units (one commit per dependency layer).
- **Blocked by test failures**: Changes cascade through many files — late-stage
  test failures may have complex root causes. Mitigation: run tests after each
  dependency layer, not just at the end.

## Implementation Notes

- Organize the branch as a clean sequence of commits following the dependency
  chain, so reviewers can verify layer by layer.
- Run `deno task test` after each layer to narrow down the source of any
  failures.
- The migration is generated as a single `drizzle-kit generate` at the end,
  after all schema.ts changes are finalized.
