# ADR-004: Delete+Regenerate Migration Strategy

## Status

Accepted

## Date

2026-07-01

## Context

The Database Reliability Standardization changes every table definition in
`db/schema.ts` — PK types, FKs, timestamps, and indexes. The project is in dev
stage with no production data: the database is frequently recreated (docker
compose down + up), and the existing migration (`0000_high_franklin_storm.sql`)
reflects the current schema before changes.

Two approaches exist: keep the existing migration and add a new 0001 migration
with ALTER TABLE statements, or delete the existing migration and regenerate
a single fresh CREATE migration.

## Decision

Delete the existing `db/migrations/0000_high_franklin_storm.sql` (and its
`meta/` snapshot), then run `drizzle-kit generate` to produce a single fresh
migration that captures the final schema in one CREATE statement.

## Alternatives Considered

### Sequential migration (0000 → 0001)

- **Description**: Keep the existing 0000 migration, make schema changes, run
  `drizzle-kit generate` to produce 0001 with ALTER TABLE (ADD COLUMN, DROP
  DEFAULT, etc.).
- **Pros**: Preserves migration history; follows standard additive migration
  convention.
- **Cons**: The 0001 migration would contain complex ALTER statements (changing
  column types from text to uuid requires DROP DEFAULT → ALTER TYPE → ADD
  DEFAULT; recreating indexes is noisy); no production data means these are
  unnecessary gymnastics; two migration files for what is conceptually one
  atomic schema state.
- **Why rejected**: The user explicitly preferred delete+regenerate at dev stage.
  The cost of maintaining migration history (two files, ALTER cruft) outweighs
  the benefit when the DB is frequently recreated.

## Consequences

### Positive

- Single atomic migration — one CREATE statement, easy to understand
- No ALTER TABLE gymnastics or type-coercion complexity
- Clean git diff: old migration removed, new migration added
- Schema.ts and migration SQL are always in sync (generated from same source)

### Negative

- Migration history is lost (one snapshot instead of two)
- If the project later gains production data, this strategy will not be viable
  (ALTER TABLE migrations will be required)
