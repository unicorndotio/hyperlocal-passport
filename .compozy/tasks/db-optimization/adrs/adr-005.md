# ADR-005: Test DB Utility with useDatabase() Hooks

## Status

Accepted

## Date

2026-07-01

## Context

Every integration test file in the project repeats the same boilerplate:

```ts
if (Deno.env.get('PG_CONNECTION')) {
  Deno.test({
    name: '...',
    sanitizeOps: false,
    sanitizeResources: false,
    fn: async () => {
      // manual insert setup...
      try {
        // test body...
      } finally {
        // manual delete teardown...
      }
    },
  })
} else {
  Deno.test({
    name: '... - Skipped (PG_CONNECTION not set)',
    fn: () => { console.info('skipped') },
  })
}
```

There are ~40 test files, all following this pattern. The PRD (F7) mandates
a shared utility to reduce duplication.

## Decision

Create `lib/test-db.ts` exporting a `useDatabase()` function that:

1. Applies the PG_CONNECTION environment variable guard (skips tests when unset)
2. Automatically sets `sanitizeOps: false` and `sanitizeResources: false` on all
   wrapped tests
3. Provides an optional `cleanupDatabase()` helper that truncates all app tables
   in FK-safe order (respecting cascade dependencies), so tests can clean up
   without manual delete chains

The API is additive — existing test files can migrate incrementally without
breaking.

## Alternatives Considered

### Minimal inline helper

- **Description**: Export a small `wrapTest()` function that only sets the PG
  guard + sanitize flags, leaving data management to individual tests.
- **Pros**: Simpler to implement; less abstraction.
- **Cons**: Tests still manage their own TRUNCATE/deletion; each test file
  duplicates the PG guard pattern.
- **Why rejected**: The user explicitly preferred the richer `useDatabase()`
  pattern with automatic setup/teardown hooks.

### No utility (status quo)

- **Description**: Keep the existing boilerplate in all ~40 test files.
- **Pros**: No new abstraction to learn; no migration cost.
- **Cons**: PRD requirement F7 explicitly calls for this utility; the
  duplication is already a maintenance burden (e.g., adding a new column to a
  table requires updating every test file's delete chain).
- **Why rejected**: PRD requirement + maintenance burden.

## Consequences

### Positive

- Eliminates boilerplate from ~40 test files
- Single point of change for test DB configuration (e.g., adding connection pool
  settings)
- `cleanupDatabase()` removes manual delete-ordering chains

### Negative

- One new file and one new import pattern to learn
- Test files must be migrated individually (cannot batch-automate safely)
