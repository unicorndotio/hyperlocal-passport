# ADR-004: Feed Route, Navigation Architecture, and Passport Island

## Status

Accepted

## Date

2026-06-29

## Context

The resident frontend needs three architectural decisions resolved:
1. Where the feed page should live in the route hierarchy
2. Whether the duplicated bottom navigation should be extracted into a shared component
3. How the passport cover animation should be implemented

## Decision

### 1. Feed Replaces Root Route

The feed becomes the root route (`/`), replacing the existing Fresh boilerplate index page. The feed is the primary discovery surface and the first thing residents see. The catalog moves to `/catalog` (unchanged). This gives the feed the canonical URL and shortest path.

### 2. Bottom Nav Extracted to Shared Component

Create `components/BottomNav.tsx` as a shared UI component (not an island — no client-side interactivity needed). It accepts an `active` prop to highlight the current tab. All three resident-facing pages (`/`, `/catalog`, `/passaporte`) use it. This eliminates the current triple-duplication of the nav markup.

### 3. PassportCover Island

Create a dedicated `PassportCover` island that handles:
- A `closed` state showing the deep blue passport cover card
- A tap/click handler that toggles to the `open` state using `useSignal`
- CSS transitions (`transform`, `opacity`) for the slide-and-fade animation
- Rendering the inner content (active redemptions + QR codes via existing `QRCodeDisplay` island) when open
- A `locked` state variant for pending/rejected residents

## Alternatives Considered

### Alternative 1: Bottom Nav as Island

- **Description**: Make the bottom nav a Preact island with client-side active-state management.
- **Why rejected**: No client-side interactivity needed — the active tab is determined at render time from the current route URL. A static component is simpler and avoids unnecessary JavaScript.

### Alternative 2: Passport Animation in Page Handler

- **Description**: Keep all passport logic in the page handler as plain CSS classes toggled via a link/button, no island needed.
- **Why rejected**: The animation state (open/closed) requires client-side interactivity. An island is the correct Fresh pattern for this. The island boundary also cleanly separates the passport cover visual logic from the page data fetching.

## Consequences

### Positive

- Clean route hierarchy: `/` = feed, `/catalog` = businesses, `/passaporte` = passport
- Reduced duplication: bottom nav markup defined once
- Clear separation: `PassportCover` island encapsulates all passport animation and locked-state logic

### Negative

- Existing `/` boilerplate page must be replaced
- Existing pages (`catalog.tsx`, `passaporte.tsx`) need minor edits to import the shared `BottomNav` instead of inline nav

### Risks

- Low: route change and nav extraction are straightforward refactors
- **Mitigation**: verify all three pages render the nav correctly after extraction
