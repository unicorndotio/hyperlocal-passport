# ADR-002: V1 Resident Frontend Product Scope

## Status

Accepted

## Date

2026-06-26

## Context

The V1 Resident Frontend scope was refined through three clarifying questions covering navigation model, merchant publishing, and passport visual polish. The selected approach determines the breadth and depth of features delivered in the initial release.

Key decisions that were confirmed:
1. Navigation: feed-first model with the hybrid feed as the home page and catalog as a secondary tab
2. Merchant posts: included in V1 as a core engagement driver
3. Passport: full Bento-style passport cover with 2D animated transition, matching the Brazilian passport aesthetic

## Decision

Adopt the **Full V1 Resident Experience** approach. Build all three pillars (feed, passport, catalog) with the full feature set and visual polish in a single V1 release. This includes:

- Feed-first navigation with the hybrid feed as the resident home page
- Hybrid feed content: automated system events (coupon releases, savings notices, admin announcements) interspersed with merchant-authored text + image posts
- Merchant post publishing tool for verified business accounts
- Full Bento-style passport cover with deep blue gradient and 2D slide-and-fade transition revealing QR code, status, and active redemptions
- Catalog as a secondary bottom-nav tab
- Verification gating for pending/rejected residents
- Online QR validation with the existing `POST /api/transactions/validate` flow

## Alternatives Considered

### Alternative 1: Progressive Build

- **Description**: Launch V1 with feed-first navigation (system-generated posts only) and minimal passport styling. Defer merchant posts and passport animation to Phase 2.
- **Pros**: Faster initial ship, simpler moderation pipeline, lower upfront build cost
- **Cons**: Feed risks feeling empty without merchant content; passport lacks the premium identity that differentiates the product; requires rework in Phase 2
- **Why rejected**: The differentiating value of Passaporte Local comes from both the merchant-resident connection and the physical passport metaphor. Delaying either weakens the V1 value proposition.

### Alternative 2: Feed-Focused V1

- **Description**: Build the full feed (system + merchant posts) with feed-first navigation, but keep the passport as a flat functional list of redemptions with QR codes.
- **Pros**: Delivers discovery and merchant marketing value in V1; functional passport is sufficient for checkout
- **Cons**: The passport is the namesake feature — shipping it without the premium experience undermines brand identity; visual polish deferred to Phase 2 risks never happening
- **Why rejected**: The passport cover is the core emotional hook for residents. A flat list of codes does not communicate "premium local benefits" and weakens first impressions.

## Consequences

### Positive

- **Complete V1 story**: residents register, discover merchants in the feed, redeem coupons, and present a beautiful passport at checkout — all in one release
- **Strong brand identity**: the passport cover animation establishes the premium feel that differentiates from generic coupon apps
- **Merchant engagement**: merchant posts provide a direct marketing channel and prevent a ghost-town feed
- **Feed-first discovery**: positions the product as a community engagement tool, not just a directory

### Negative

- **Higher V1 scope**: merchant publishing pipeline, image compression, moderation, and passport animation all must be built before launch
- **Content moderation overhead**: merchant posts introduce a risk of low-quality or inappropriate content
- **Feed performance risk**: merchant image uploads could impact loading times if not properly optimized
- **Animation accessibility**: the 2D transition must work across browsers and assistive technologies

### Risks

- **Merchant adoption risk**: if merchants don't post, the feed relies solely on system events
- **Mitigation**: seed with system-generated events (new coupons, transaction savings) and build a low-friction publishing UI to minimize merchant effort
- **Animation performance risk**: 2D transitions may stutter on low-end devices
- **Mitigation**: use hardware-accelerated CSS properties (transform, opacity) and test on target devices
