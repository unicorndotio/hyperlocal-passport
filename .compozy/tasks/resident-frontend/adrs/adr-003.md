## Context

The resident hybrid feed needs to display two content types in a single reverse-chronological stream: system-generated events (coupon releases, transaction savings notices, admin announcements) and merchant-authored posts (text + optional image). We need to decide how to model this data for efficient querying and pagination.

The feed is the primary discovery surface and home page of the app, so query performance and developer maintainability are key concerns.

## Decision

Use a **union query over existing tables** rather than creating a unified `feed_events` table. The feed handler will:

1. Query multiple sources: a new `merchant_posts` table for merchant content, the existing `coupons` table for coupon release events, the existing `transactions` table for savings notices, and potentially other sources for admin announcements
2. Map each row to a virtual `FeedEvent` type with a `type` discriminator and `createdAt` sort key
3. Merge and sort in-memory (or via SQL `UNION ALL` with ordering), then paginate

This avoids dual-write problems (writing to both a business table and a feed table for every event) and keeps the data model normalized.

## Alternatives Considered

### Alternative 1: Unified `feed_events` Table

- **Description**: Create a single `feed_events` table with `type`, `businessId`, `content` (JSONB), and `createdAt`. Every event (coupon creation, transaction, merchant post) writes a row to this table.
- **Pros**: Simple single-table query; easy to paginate; consistent ordering
- **Cons**: Dual-write required for every event (write to source table + feed table); risk of data inconsistency; more moving parts in transactions
- **Why rejected**: Adds write complexity and sync risk without sufficient benefit. The union query approach handles the same requirement without dual-writes.

### Alternative 2: Materialized View

- **Description**: Create a PostgreSQL materialized view that unions the sources and refresh it periodically.
- **Pros**: Clean SQL; no in-memory merging
- **Cons**: Stale data between refreshes; refresh overhead; over-engineering for V1 feed volumes
- **Why rejected**: Feed needs real-time freshness; materialized views add operational complexity.

## Consequences

### Positive

- **No dual-write risk**: each event is written once to its authoritative table
- **Normalized schema**: no data duplication across tables
- **Flexible evolution**: new event types can be added by adding a new source query

### Negative

- **In-memory merge**: the handler must fetch from multiple tables and merge/sort results
- **Pagination complexity**: cursor-based pagination requires a virtual sort key across heterogeneous sources
- **Query count**: multiple queries per feed load (mitigated by batching and indexing on `createdAt`)

### Risks

- **Performance at scale**: feed page could slow down as transaction volume grows
- **Mitigation**: add composite indexes on `createdAt` for each source table; limit page size to 20 items; monitor query latency
