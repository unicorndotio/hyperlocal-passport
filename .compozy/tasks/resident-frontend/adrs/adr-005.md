# ADR-005: Feed Data Model — Materialized View for Global Content with User-Specific Query

## Status

Accepted

## Date

2026-06-29

## Context

The resident feed must display two categories of content with different access scopes:

1. **Global content**: merchant posts, coupon release announcements, and admin announcements — all residents see the same items
2. **User-specific content**: transaction savings notices — either personalized ("You saved R$ 20") or anonymized community-wide notices

The initial design proposed a union query across multiple tables merged in-memory by the handler. However, this creates complex cursor-based pagination across heterogeneous sources and risks performance degradation as transaction volume grows.

A PostgreSQL materialized view offers single-table query performance for the global portion, at the cost of some staleness. The user-specific portion requires a separate query regardless.

## Decision

Use a **hybrid approach**: a materialized view for global feed content, plus a separate user query for transaction-based savings notices.

### Materialized View

Create a `feed_events` materialized view that unions:

- `merchant_posts` (where `is_visible = true`)
- `coupons` (newly created/active — coupon release events)
- Admin announcements (future table or mechanism)

The view includes a `type` discriminator, `created_at` sort key, and all fields needed for rendering. It uses `UNION ALL` with `ORDER BY created_at DESC`.

### Refresh Strategy

Use `REFRESH MATERIALIZED VIEW CONCURRENTLY` triggered on-write from the API handlers that create the source data:

- After `POST /api/posts`: refresh the MV
- After coupon creation (existing endpoint): refresh the MV
- After admin announcement creation: refresh the MV

This keeps staleness under 500ms per event while avoiding periodic polling.

### Feed Handler Query

The feed handler (`GET /api/feed`) queries the MV for global content with cursor-based pagination, then runs a separate query against `transactions` for the current user's savings notices. Results are merged in-memory by `created_at`.

## Alternatives Considered

### Alternative 1: Pure Union Query (In-Memory Merge)

- **Description**: Handler queries each source table independently and merges results in-memory
- **Pros**: Real-time data; no refresh mechanism; no MV maintenance
- **Cons**: Complex cursor pagination across heterogeneous sources; multiple queries per request; no index on virtual sort key
- **Why rejected**: Pagination complexity outweighs benefits for V1. The MV simplifies the 90% case (global content).

### Alternative 2: Unified Materialized View for Everything (Including Transactions)

- **Description**: Include transaction savings in the MV as anonymized global events
- **Pros**: Single query for all content; simplest handler
- **Cons**: Cannot personalize savings notices per user; loses the "your savings" context; stale transaction data
- **Why rejected**: PRD goals include showing residents their personal savings. The MV cannot support user-specific data.

## Consequences

### Positive

- **Simple pagination**: cursor-based pagination on the MV works with a single `created_at` index
- **No in-memory merge for global content**: one well-indexed query serves the feed
- **Personalization preserved**: user-specific transaction data queried separately
- **Normalized writes**: no dual-write — source tables remain authoritative; the MV is a read-only cache

### Negative

- **Staleness window**: up to ~500ms between event creation and MV refresh
- **MV maintenance**: requires unique index for CONCURRENTLY refresh; refresh adds write latency to post/coupon creation handlers
- **Refresh-on-write coupling**: post/coupon creation handlers must know about the MV

### Risks

- **MV refresh contention**: concurrent writes could contend on MV refresh
- **Mitigation**: `REFRESH MATERIALIZED VIEW CONCURRENTLY` uses a lock that allows concurrent reads; test under load
- **MV refresh failure**: if refresh fails, the MV becomes stale
- **Mitigation**: log refresh errors; add a periodic `REFRESH` fallback (every 60 seconds)
