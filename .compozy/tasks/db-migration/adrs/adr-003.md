# ADR-003: Docker Compose PostgreSQL with Drizzle Gateway Infrastructure

## Status

Accepted

## Date

2026-06-19

## Context

The current deployment is a single Docker container running the Deno Fresh application with file-backed Deno KV. Adding PostgreSQL requires a database server, and Drizzle Gateway provides the web UI for browsing and editing data. The infrastructure must work identically in development and production to minimize environment-specific bugs.

Key requirements:
- PostgreSQL must be healthy before the web app starts
- Drizzle Gateway needs persistent storage for its configuration
- The Drizzle Gateway UI must be accessible but protected by a master password
- All services must work within a single Docker Compose file for simplicity
- File uploads continue to use local filesystem volumes (unchanged)

## Decision

Add three services to docker-compose.yml alongside the existing web service:
1. **PostgreSQL** (`postgres:18-alpine`) — database server with health check, persistent volume, and initial database setup
2. **Drizzle Gateway** (`ghcr.io/drizzle-team/gateway:latest`) — web UI on port 4983, protected by MASTERPASS environment variable, with persistent volume for its store
3. **Web app** — updated to depend on PostgreSQL (healthy) and use `PG_CONNECTION` environment variable instead of `DENO_KV_PATH`

## Alternatives Considered

### Alternative 1: Managed PostgreSQL (Neon/Supabase) + Local Drizzle Gateway

- **Description**: Use a cloud PostgreSQL provider (Neon, Supabase) for both dev and prod. Drizzle Gateway connects to the cloud database.
- **Pros**: No local PostgreSQL container to manage, automatic backups, point-in-time recovery.
- **Cons**: Internet dependency for local development, latency on every query, vendor lock-in, cost at scale. Complicates offline development.
- **Why rejected**: Docker Compose PostgreSQL is simpler for the current single-server deployment. Can migrate to managed PostgreSQL later if scaling requires it.

### Alternative 2: PostgreSQL Only, No Drizzle Gateway

- **Description**: Add PostgreSQL container but skip Drizzle Gateway. Developers use `psql` CLI or pgAdmin for data inspection.
- **Pros**: Fewer containers, smaller resource footprint, one less service to maintain.
- **Cons**: No convenient web UI for data browsing. Admin team would need to install pgAdmin or use CLI, creating friction for support scenarios.
- **Why rejected**: The user explicitly identified lack of data UI as a pain point. Drizzle Gateway solves this directly with minimal operational cost.

## Consequences

### Positive

- Production-like database environment in development
- Drizzle Gateway provides a web UI for all data browsing and editing
- Consistent infrastructure across dev and prod — no surprises at deploy time
- PostgreSQL health check ensures app never starts without a database
- MASTERPASS authentication secures the Gateway from unauthorized access
- File uploads continue unchanged on local filesystem volumes

### Negative

- Three containers instead of one increases resource usage
- Docker Compose startup is slower (must wait for PostgreSQL health check)
- Drizzle Gateway pulls from GitHub Container Registry, which may require authentication
- PostgreSQL data volume needs backups (not automated in this setup)

### Risks

- **Drizzle Gateway image authentication**: The `ghcr.io/drizzle-team/gateway:latest` image may require Docker login to ghcr.io. *Mitigation:* Document the `docker login ghcr.io` step in setup instructions, same as the user's existing pattern.
- **PostgreSQL memory usage**: Alpine PostgreSQL with default config may use significant memory. *Mitigation:* Set `PGOPTIONS='-c shared_buffers=256MB'` if needed, or accept default for V1.
- **Volume path management**: The compose file references `${VOLUME_PATH}` which must be set in the environment. *Mitigation:* Provide a `.env.example` file with documented defaults.

## Implementation Notes

- Use the latest PostgreSQL image (`postgres:18-alpine`).
- PostgreSQL credentials (user, password, database) configured via environment variables.
- Drizzle Gateway `STORE_PATH` should be a persistent volume so configuration survives restarts.
- The web app's connection string format: `postgresql://root:password@postgres:5432/pg`.
- Watchtower is not needed in V1 — defer to a future infrastructure PRD.
