# ADR-009: Coupon Views Tracking — Aggregated Counter Only for V1

## Status

Accepted

## Date

2026-06-19

## Context

Coupon views are currently tracked as KV atomic counters under `['analytics', couponId, 'views']`. The counter is incremented each time a business page is loaded (in the GET handler for `routes/business/[id].tsx`). There is no individual view event record — just the running total.

The PRD analytics dashboard shows "views" as a simple number in the per-coupon funnel. Views are approximated (no deduplication — page refreshes count as multiple views). The current behavior is acceptable for V1.

## Decision

Store coupon views as a single integer column (`views`) in the `coupon_analytics` table, incremented atomically via SQL `UPDATE coupon_analytics SET views = views + 1 WHERE coupon_id = ?`. No individual view event log table is created. The view counter is incremented directly (not wrapped in a wider transaction, matching current behavior where view counts are fire-and-forget).

## Alternatives Considered

### Alternative 1: Both Counter + Individual View Events

- **Description**: A `coupon_views` event table with columns `(id, coupon_id, user_id [nullable], viewed_at)` that records every page view, plus the aggregated counter in `coupon_analytics`.
- **Pros**: Enables future analytics features (unique viewers, time-of-day analysis, view trends over time).
- **Cons**: Every page view writes two rows (event + counter update). The event table grows unboundedly — even at V1 scale, a single page refresh generates a row. Doubles write cost per view. YAGNI violation.
- **Why rejected**: No requirement for per-view analytics in the PRD. The event log can be added when analytics drill-down features are explicitly requested.

## Consequences

### Positive

- Minimal schema: single integer column in existing `coupon_analytics` table
- Fast atomic increment (no read-before-write, no contention on SELECT)
- Same behavior as current KV counter — no behavioral change
- No unbounded-growth table to manage

### Negative

- Cannot distinguish unique views from total views (page refreshes = additional count)
- No view timing data (can't answer "when do most views happen?")
- No per-user view tracking

### Risks

- **None significant**: The counter behavior is identical to the current KV implementation. If detailed view analytics are needed later, a `coupon_views` event table can be added and backfilled from access logs (if available) or started fresh.

## Implementation Notes

- Counter increment: `await db.update(schema.couponAnalytics).set({ views: sql`${schema.couponAnalytics.views} + 1` }).where(eq(schema.couponAnalytics.couponId, couponId))`
- First view for a coupon: UPSERT pattern — if no analytics row exists, INSERT with `views = 1`
- Increment happens in the business page GET handler (no transaction wrapping — fire and forget, matching current behavior)
- Coupon analytics row is created lazily on first view (no pre-creation at coupon creation time)
