# ADR-001: Full Database Migration from Deno KV to PostgreSQL with Drizzle ORM

## Status

Accepted

## Date

2026-06-19

## Context

The Passaporte Local platform is currently built on Deno KV as its sole database. During development, several issues emerged:

1. **Deno KV is unstable** — The `--unstable-kv` flag is required, meaning the API may change or be removed in future Deno versions. It is explicitly labeled as not production-ready.
2. **No query capabilities** — KV is a key-value store. Complex queries, joins, aggregations, and filtering require manual indexing or O(N) scans, which become increasingly problematic as the data model grows.
3. **No data UI** — There is no way to browse, inspect, or edit data directly. Debugging data issues requires raw KV key inspection.
4. **Better Auth KV adapter maintenance burden** — A custom KV adapter maintains indexes manually. Every new indexed field requires code changes. The adapter must handle atomic multi-key operations, stale index cleanup, and fallback O(N) scans.
5. **Proximity to customer features** — The next development phase (end-user/customer features) will add more data complexity (orders, preferences, notifications, etc.). Continuing on KV would compound technical debt at a critical growth point.
6. **No migration tooling** — KV has no schema migrations. Schema changes are manual data transforms, making team collaboration risky.

The Drizzle ORM ecosystem (PostgreSQL driver, Drizzle Kit for migrations, Drizzle Gateway for data UI) provides a production-ready, type-safe alternative with migration management and a built-in database browser.

## Decision

Migrate all data from Deno KV to PostgreSQL using Drizzle ORM in a single coordinated effort ("Big Bang"). This includes:

- All business data: users (residents/businesses/admins), businesses, coupons, redemptions, transactions, demand signals
- All infrastructure data: Better Auth tables (user, session, account), analytics counters, file metadata, rate limiter counters, approval queues
- Replace the custom Better Auth KV adapter with Better Auth's native Drizzle adapter
- Define every table schema in a single centralized `src/db/schema.ts` file
- Generate one initial migration that creates all tables
- Drop the old KV data file (no data migration since there are no production users)
- Add PostgreSQL and Drizzle Gateway as Docker Compose services alongside the web app

## Alternatives Considered

### Alternative 1: Two-Phase Migration

- **Description**: Migrate app data models first (businesses, coupons, redemptions, transactions, signals), keeping analytics counters and rate limiters on KV. Migrate those in a second phase.
- **Pros**: Smaller initial blast radius, faster first PR.
- **Cons**: Two migration cycles, dual-database complexity during transition, maintenance burden of keeping KV alive for a subset of data, analytics counters are simple and would still need migration eventually.
- **Why rejected**: The analytics counters (views, redemptions, validations) are tightly coupled to every coupon redemption and transaction. Splitting them adds atomicity complexity. A single migration is faster and simpler for a pre-launch project.

### Alternative 2: Micro-Migration by Domain

- **Description**: Migrate one domain at a time: businesses + signals first, then coupons + redemptions, then auth, then analytics. Each domain gets its own migration, PR, and test cycle.
- **Pros**: Lowest risk per change, easiest to debug if something breaks.
- **Cons**: Highest overhead — 4+ migration cycles, each requiring schema coordination, test updates, and potential data sync. Would drag the timeline by weeks. Overkill for a project with no production users.
- **Why rejected**: The project is pre-launch with no real users. There is no production data to protect. The "blast radius" argument doesn't apply when the entire database can be dropped and recreated safely.

## Consequences

### Positive

- Single source of truth for database schema in TypeScript
- Type-safe queries with full IDE autocompletion
- Migration framework for future schema changes (additive, reversible)
- Drizzle Gateway provides a web UI to browse and edit data directly
- Better Auth Drizzle adapter is officially maintained, reducing custom code
- PostgreSQL supports complex queries, joins, aggregations needed for customer features
- No `--unstable-kv` flag required for production deployment

### Negative

- Additional infrastructure dependency (PostgreSQL container, Drizzle Gateway container)
- Higher memory and disk footprint compared to file-backed KV
- Increased Docker Compose complexity (3 services instead of 1)
- All existing route handlers and islands must be updated from `kv.get/set` to Drizzle queries
- Test infrastructure must switch from in-memory KV to test PostgreSQL (or use Drizzle's test utilities)

### Risks

- **Migration scope is large** — Every route handler, test file, and island touches KV. Missed a KV call? It will 500 at runtime. *Mitigation:* Systematic audit of all `kv.` references before deployment, plus comprehensive test suite updates.
- **Docker Compose complexity** — Three services with health checks and dependency ordering. *Mitigation:* Follow the user's proven docker-compose pattern from another project (PostgreSQL health check, Drizzle Gateway depends on healthy PostgreSQL).
- **Drizzle Kit + Deno compatibility** — `drizzle-kit` relies on Node.js APIs. *Mitigation:* Run `drizzle-kit` via `npx` or install globally with npm and invoke as a Deno task. The ORM runtime (`drizzle-orm`) works fine with Deno via npm specifier.

## Implementation Notes

- Drizzle Kit CLI may need to run outside Deno (via `npx` or global npm install). The ORM runtime (`npm:drizzle-orm/postgres-js`) works within Deno.
- The `lib/kv.ts` file and `lib/kv-adapter.ts` will be deleted after migration.
- The Better Auth Drizzle adapter replaces the custom KV adapter in `lib/auth.ts`.
- All test files using in-memory KV must switch to database test patterns (testcontainers or transaction-rollback per test).
- File storage (`lib/storage.ts`) stays on local filesystem — only the file metadata moves to PostgreSQL.
- Analytics counters (views, redemptions, validations) replace KV atomic increments with SQL `COUNT` queries or a dedicated analytics table.
