# ADR-004: Drizzle Client Connection Strategy — Single Pool with pg Driver

## Status

Accepted

## Date

2026-06-19

## Context

The Fresh app handles concurrent requests from multiple users simultaneously. The current Deno KV approach uses a single shared `Deno.openKv()` instance, which handles concurrency internally. With PostgreSQL, a single connection serializes all database queries — a slow query would block all others. A connection pool manages multiple concurrent connections efficiently.

## Decision

Use `pg.Pool` (from `npm:pg`) as the PostgreSQL driver with a configurable minimum/maximum connection pool, wrapped by Drizzle ORM. The pool is created once at startup and shared across the application.

## Alternatives Considered

### Alternative 1: Single PostgreSQL Client

- **Description**: A single `pg.Client` connection used for all queries.
- **Pros**: Simplest setup, minimal resource usage.
- **Cons**: Serializes all database queries. One slow query blocks all others.
- **Why rejected**: Fresh handles concurrent requests; a single connection would become a bottleneck at the first sign of traffic.

### Alternative 2: PgBouncer Connection Pooler

- **Description**: Add a PgBouncer container between the app and PostgreSQL for connection pooling.
- **Pros**: Handles very high connection counts efficiently, connection reuse.
- **Cons**: Additional service to deploy and configure. Overkill for V1 scale (single neighborhood, ~50 businesses).
- **Why rejected**: `pg.Pool` is sufficient for V1. PgBouncer can be added later if scaling requires it.

## Consequences

### Positive

- Concurrent requests can execute database queries in parallel
- Pool size is configurable via environment variable (default: min=1, max=10)
- Standard Drizzle + Deno pattern — well-documented and supported
- Pool manages connection lifecycle (reconnect on failure, cleanup on drain)

### Negative

- Slightly more complex setup than a single connection
- Connection pool consumes PostgreSQL server resources (each connection uses ~10MB)

### Risks

- **Pool exhaustion under high concurrency**: If all pool connections are in use, new requests queue up. *Mitigation:* Start with max=10, monitor with logs, increase if needed. Add PgBouncer in Phase 2 if sustained high concurrency.
- **Connection leaks**: If a query fails to release its connection, the pool drains. *Mitigation:* Use Drizzle's transaction and query methods which automatically release connections.

## Implementation Notes

- Pool configuration: `min: 1, max: 10, connectionString: Deno.env.get("PG_CONNECTION")`
- Drizzle client creation: `drizzle({ client: new Pool({ connectionString }), schema })`
- Pool is created in `lib/db.ts` and exported as the single `db` instance
- No connection pooling middleware — Drizzle handles release automatically
