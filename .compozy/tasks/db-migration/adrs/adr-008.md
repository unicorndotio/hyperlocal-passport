# ADR-008: User Coupon Usage — SQL COUNT from Redemptions Table

## Status

Accepted

## Date

2026-06-19

## Context

The current codebase tracks monthly coupon usage per user via a KV counter key: `['user_coupon_monthly_count', userId, couponId, yearMonth]`. This counter is incremented atomically on each redemption and checked before allowing a new redemption (e.g., "max 2 per month per user per coupon").

With PostgreSQL, redemptions are stored as rows in a `redemptions` table. Each redemption already records `user_id`, `coupon_id`, and `redeemed_at` timestamp. The monthly usage count can be computed directly from this table.

## Decision

Eliminate the separate `user_coupon_monthly_count` counter. Replace both the check and increment with a single approach: query the `redemptions` table using `SELECT COUNT(*) WHERE user_id = ?, coupon_id = ?, redeemed_at >= date_trunc('month', NOW())`. The check and the INSERT happen in the same database transaction to prevent race conditions.

## Alternatives Considered

### Alternative 1: Dedicated `coupon_usage` Table

- **Description**: A separate table `coupon_usage(user_id, coupon_id, usage_date)` with one row per redemption per user per coupon.
- **Pros**: Explicit tracking, supports queries beyond monthly counts (e.g., lifetime usage per user).
- **Cons**: Redundant with the `redemptions` table which already stores this data. Two tables tracking the same events increases maintenance burden and risk of drift.
- **Why rejected**: The `redemptions` table already contains all needed information. A separate table is redundant.

### Alternative 2: Counter Column on a Join Table

- **Description**: A table `user_coupon_stats(user_id, coupon_id, year_month, monthly_count)` with UPSERT, mirroring the current KV pattern in a relational table.
- **Pros**: Closest translation of the current KV approach. O(1) read for the monthly count.
- **Cons**: Requires an UPSERT on every redemption in addition to the redemption INSERT. Redundant data that can become out of sync.
- **Why rejected**: SQL COUNT on the `redemptions` table is trivially fast with proper indexing and eliminates the redundancy and sync risk.

## Consequences

### Positive

- Single source of truth: the `redemptions` table
- No redundant counter to maintain or reconcile
- One less table in the schema
- Proper indexing on `(user_id, coupon_id, redeemed_at)` makes COUNT queries efficient

### Negative

- COUNT query takes longer than a direct counter lookup (milliseconds vs microseconds) — acceptable at V1 scale
- Monthly boundary is computed by the query — slightly more complex than reading a pre-computed value

### Risks

- **COUNT performance at scale**: With millions of redemptions per user per coupon, COUNT queries become slow. *Mitigation:* The index on `(user_id, coupon_id, redeemed_at)` ensures the COUNT is a simple index scan. If performance becomes an issue, a materialized view or counter can be added later.
- **Race condition on monthly limit check**: Two concurrent redemption requests could both check the count, see 1 (limit=2), and both proceed. *Mitigation:* The check and INSERT are in the same serializable transaction. PostgreSQL row-level locking ensures consistency.

## Implementation Notes

- Index: `CREATE INDEX idx_redemptions_user_coupon_month ON redemptions(user_id, coupon_id, redeemed_at);`
- Drizzle query: `db.select({ count: count() }).from(redemptions).where(and(eq(redemptions.userId, userId), eq(redemptions.couponId, couponId), gte(redemptions.redeemedAt, startOfMonth)))`
- The redemption INSERT and the count check are wrapped in a Drizzle transaction (replacing the current `kv.atomic()`)
- Current code in `redeem.ts` that checks MonthlyRedeemCount then increments is replaced
